## Core Principles — Review Discipline

- **[2026-05-08] When changing a contract on shared auth/fence code, audit ALL callers AND all assertions in their tests in the same patch — and verify the change is necessary at all.** B.SWEEP changed orphan-fence's `lookup_error → throw` to `lookup_error → redirect('/onboarding')` to "self-heal a transient missing-row case," but `.maybeSingle()` already routes truly-missing rows through the `orphan` branch (`data:null, error:null`). The contract change was a misdiagnosis: it didn't fix a real bug AND it broke the C1-B forged-cookie regression guard (RLS-denial errors masquerading as orphan → silently land on `/onboarding` instead of `/login`). Only 4 page-guard tests were re-aligned; the forged-cookie E2E was deferred to followups. Discipline: (a) before changing a shared-helper contract, prove the new behavior is needed (grep for the symptom path, write the failing test that the OLD contract can't satisfy); (b) audit every consumer + every assertion across the test tree (`grep -rn "<old-contract-marker>" tests/`) in the SAME patch — not "deferred to followups"; (c) if the symptom is already handled by an adjacent branch (here: `orphan` branch handles missing rows via maybeSingle's null-data path), revert the contract change rather than ship it. The fix that closes the gap is often the revert + a narrow defense-in-depth carveout (here: PGRST116-only redirect inside `lookup_error` for the rare edge case where Supabase returns no-row as an error rather than null data) — preserves the original contract while honoring the original "missing row → onboarding" intent. Pairs with line 134 (Sibling path symmetry) but applies one layer up: before fixing siblings, verify the original contract change was needed at all. (source: bugfix-tomi · project: kalori)
- **[2026-05-08] When a bugfix-tomi batch ships design-change work alongside true bugs, the per-bug Codex review at Phase 4 must explicitly verify that any `Planning/ui-design.md` (or equivalent design-doc) edits land ATOMICALLY in the same commit as the implementation that ratifies them.** Bug #4 in the mobile-ui-overhaul batch self-added a tiebreaker (#23) to `Planning/ui-design.md` to justify expanding scope to LibraryTab's quantity-picker; the resume sub-agent verified the proposal had cited LibraryTab so the addition was warranted, but the pattern of self-justification-via-design-doc-edit is a known scope-creep risk — code that ships without the matching design-doc commit creates an undocumented pattern; design-doc that lands without the matching code creates an aspirational tiebreaker. Discipline: at Phase 4 Codex framing, name "design-doc edits MUST appear in the same commit-set as the implementation; flag any solitary `Planning/ui-design.md` change OR any new design pattern landing without a corresponding tiebreaker entry." This catches both directions: scope creep (tiebreaker self-added to ratify expansion the user didn't ask for) AND silent precedent (new design pattern shipped without paper trail). Pairs with the existing AC-text-vs-impl-reality lessons (lines 60, 164) — those are about ACs vs code; this is about design-doc vs code. (source: bugfix-tomi · project: kalori)
- **[2026-05-08] `startTransition` does NOT commit `setBusy(true)` synchronously — same-tick double-submit bypasses the React-state guard.** A submit handler that reads `if (busy) return` at the top, then calls `setBusy(true)` inside `startTransition(() => ...)`, ships a re-entrancy hole: two synchronous `requestSubmit()` / Enter-key fires in the same tick both observe `busy === false` and both reach the network. Pattern: `useRef<boolean>` latch checked AND set BEFORE entering any transition (`if (inFlightRef.current) return; inFlightRef.current = true; startTransition(...)`); cleared in finally + on unmount. React state is for renders, refs are for synchronous gates. Applies to ANY form using `startTransition` + an "in-flight" guard, including server-action forms. (source: superpowers-exec-tomi · project: kalori-mvp-stabilization)
- **[2026-05-08] `setTimeout` scheduled in an async success branch fires on a destroyed component unless guarded at BOTH schedule site AND callback.** Unmount cleanup that clears the *currently-pending* timer ID does NOT prevent a NEW timer being scheduled later when `await fetch()` resolves AFTER the user navigates away. The defer pattern (`onSuccess → setTimeout(() => router.refresh(), 200)`) lands `router.refresh()` on whatever route the user is on now. Pattern: `mountedRef` checked before scheduling the timer AND inside the timer callback; `useEffect` cleanup sets `mountedRef.current = false`. Same trap with `requestAnimationFrame`, `queueMicrotask`, and any post-await side effect. (source: superpowers-exec-tomi · project: kalori-mvp-stabilization)
- **[2026-05-08] E2E discovers mount-lifecycle bugs unit tests miss when listener lifetime is tied to a component subtree that unmounts before the predicate flips.** Unit tests render the listener-owning component standalone — listener subscribes, predicate flips, listener fires. Production embeds the same component inside a state-machine swap (e.g. tab/phase transitions where a sibling replaces the subtree); the listener's `useEffect` cleanup unsubscribes BEFORE the predicate flip happens elsewhere, and the rising edge is missed. Bundled E2E hits the production composition and exposes the gap. Mitigation: for any listener mounted in a component that participates in a phase/state machine, ask "what is this component's mount lifetime relative to the events I'm listening for?" before reaching for `subscribeWithSelector` / `useEffect(predicate)`. Two compatible fixes: (a) relocate the listener to a chrome-level component that stays mounted across phase transitions; (b) move the state mutation INTO the store action that flips the predicate, so the action itself is the side effect (eager mutation, no listener required). Option (b) is correct-by-construction; option (a) keeps the listener pattern but requires understanding the chrome boundary. Applies to Zustand `subscribeWithSelector`, Redux subscriptions, RxJS observables, native event listeners — any subscribe-and-fire-on-predicate pattern where lifetime depends on component mount. (source: superpowers-exec-tomi · project: kalori-mvp-stabilization)
- **[2026-05-08] When N consumers call the same helper and a contract changes, fix at the helper, not at every consumer — surgical AND surface-shrinking.** Fence-style preflights (orphan-profile fence, auth gate, feature-flag check) often live in a shared helper invoked from N pages/route handlers. When the helper's contract changes (e.g. `throw 401` → `redirect('/onboarding')`), the temptation is to update each consumer's error-handling block. Discipline: prefer one fix at the helper that mutates behavior at the source AND shrinks the surface (one branch instead of N), unless a per-consumer reason exists (different fallback UI, different telemetry tag). Why: per-consumer fixes drift over time as new consumers ship without remembering to update; helper-level fix is automatic for any future consumer. Distinct from "Sibling path symmetry" (line 132), which says fix-all-siblings-in-same-commit when the duplication can't be DRYed; this rule says DRY first, fix the surface, when the helper exists. Applies to any duplicated pattern across N callsites where a single helper would naturally own the contract. (source: superpowers-exec-tomi · project: kalori-mvp-stabilization)
- **Fixing request SHAPE is not fixing ERROR BEHAVIOR.** When a Codex Round 1 fix corrects the POST body schema, the next-round finding is often the unchanged `try/finally` that swallows server errors and hides the UI on `finally`. UI announces success after server 4xx/5xx. Test the failure branch explicitly (mock 500, assert toast stays, assert card stays visible) in the same commit as the shape fix — Round 2 will otherwise find it as a regression, not a new defect.

## Core Principles — Testing

- **[2026-05-08] False-green tests that assert PRESENCE (e.g., `getByTestId('trigger')` exists) without asserting BEHAVIOR (the trigger does its job) are technical debt with a green checkmark — Codex's adversarial framing "does this primitive serve its stated purpose?" catches them where TDD didn't.** TDD enforces "test must fail then pass," but the gap is in WHAT the test asserts: a test that mounts a component, asks for an element by testid, and verifies non-null can pass while the element does nothing user-observable when activated. C2 in the mobile-ui-overhaul batch shipped under TDD-green tests because the integration test only verified trigger render, not trigger behavior — Codex's R1 framing forced re-asking "what does this primitive PROMISE the user, and does the test exercise that promise?" Pattern at PR review: for every newly-added test that uses `getByTestId` / `getByRole` / `findBy*` without a follow-up `click()` + outcome assertion (state change, network call, focus shift, aria-live announcement), require either (a) the follow-up assertion or (b) a comment explaining why presence alone is sufficient (e.g. snapshot test, smoke test). Distinct from line 100's "live wire, dead consumer" pattern (boundary-write tests passing while consumer ignores the wire) — this is one layer earlier: the test never even probes the wire's behavior. (source: bugfix-tomi · project: kalori)
- **[2026-04-30] Tests must verify EFFECTIVE behavior, not boundary writes.** A toggle that writes localStorage + sets a `data-attr` can pass behavioral tests while the consumer hook ignores both inputs and the CSS mirror selector is missing. Pattern: integration test must read computed style / actual rendered output AFTER the toggle, not just confirm the boundary side-effects fired. Why: surface-level mocks/spies are happy when the wire is connected on one end and dangling on the other — only end-to-end effect assertions catch full disconnects. Adjacent to the dead-helper-test pattern but the failure mode is "live wire, dead consumer" not "test on orphan."
- **[2026-05-08] E2E `waitForTimeout(N)` after a fire-and-forget POST is always wrong — replace with `waitForResponse(predicate)` registered BEFORE the action.** Fixed-duration sleeps after a POST that round-trips through a cross-region pair (e.g. Vercel `iad1` ↔ Supabase `ap-southeast-1` ≈ 150–200ms one-way RTT, plus server processing) flake under contention even when the value (e.g. 500ms) is "comfortably" larger than median latency. The deterministic pattern: `const undoResponse = page.waitForResponse(r => r.url().includes('/api/...') && r.request().method() === 'POST' && r.status() === 200); await page.click(...); await undoResponse;` — registers the listener BEFORE the click so the response can't fire before `waitForResponse` is attached, and asserts the actual contract (URL match + method + status) rather than a wall-clock proxy. Same pattern beats `waitForTimeout` in any scenario where the test's logical "next step" is "wait for this server side-effect to land" — undo POSTs, optimistic-update reconciliations, mutation-then-RSC-revalidation flows. The cost is one extra line; the savings are zero flakes under cross-region latency, parallel-worker contention, or CI cold-start. Bug 2 + Bug 3 of the e2e-regressions batch were both this exact failure mode — bulk-delete-undo and single-delete-undo flaked on `waitForTimeout(500)` and went 100% green on `waitForResponse(...)` after 30 reps. (source: bugfix-tomi · project: kalori)
- **RED-first on extension to existing system = characterization, not absence.** When new spec extends a system already covering adjacent behavior, strict "must fail before code" forces inventing fake bugs. Correct discipline: assertion names the precise failure mode, only the truly-novel case lights up RED, already-covered cases pass at RED so the suite isn't a no-op. Document actual-vs-predicted RED count.

## Core Principles — Process & Sub-agents / Planning & Briefing

- **[2026-05-08] Phase 0 project-context priming that reads narrative documentation (PRD / progress.md / design-doc as text) cannot be trusted for "is feature X already shipped?" questions when documentation lags behind code; only Phase 1 sub-agents doing targeted source/migration/route reads catch divergence.** Bug #6 in the mobile-ui-overhaul batch was framed in the user's bug-list as "water-logging functionality"; Phase 0 priming reported "Task 3.5 — ZERO code shipped" based on `progress.md` text. Phase 1's investigation sub-agent went deeper — read actual `supabase/migrations/0006_*.sql`, `app/api/water/log/route.ts`, `components/dashboard/WaterTracker.tsx`, and confirmed the feature shipped end-to-end across commits b529290, 0321f01, c706d50. Bug #6 was closed as DUPLICATE of Bug #5 (Path A: water FAB navigates to existing WaterTracker). Discipline: at Phase 0 priming, mark every "shipped/unshipped" claim sourced from prose as PROVISIONAL; require Phase 1 verification against actual artifacts (migrations / route files / component files / git log of the implicated commit range) before any sub-agent acts on the priming summary. Cheap retrofit: in priming output, tag prose-sourced status claims with `[unverified-source: progress.md §X]` so downstream agents know to re-check. Applies to any bugfix-tomi or exec-tomi flow where the claimed state could be stale relative to recently-merged work. (source: bugfix-tomi · project: kalori)
- **[2026-05-08] "Delete X" framings in task cards routinely hide a substitution requirement; briefing extraction must audit the consumers of X for replacement keys.** A task labeled "delete `t.foo.stubHeading` + `t.foo.stubBody`" implicitly REQUIRES adding `t.foo.heading` if the page's `<h1>` was sourcing copy from `stubHeading` — the deletion alone leaves an undefined-text render. Briefing sub-agent must (a) grep production callsites of every to-be-deleted symbol, (b) for each callsite, decide whether the callsite is also deleted OR substituted with a NEW key, (c) surface NEW keys in the AC list even when the task card says "delete." Patch-shape holds only after this audit. Anti-pattern: faithfully implementing "delete the two keys" without the substitution, shipping a render with `undefined` text. Applies to any "remove placeholder" / "drop legacy field" / "delete deprecated alias" task where consumers may need rewiring. (source: superpowers-exec-tomi · project: kalori-mvp-stabilization)

## Domain-Specific Notes — Next.js 16 (App Router + Turbopack)

- **`useTransition().isPending` does NOT track Promise lifetime — only React rendering priority.** Wrapping `startTransition(async () => await sideEffect())` schedules the callback then drops `isPending` BEFORE the awaited Promise resolves. For UI that must reflect "request in flight" (disable retry buttons, show spinner), use explicit `useState<boolean>` set true before `await`, cleared in `finally`. Keep `useTransition` only for rendering-priority hints. (source: superpowers-exec-tomi · project: kalori)
- **React 19 `useOptimistic` state decays the instant the action resolves.** Card "fade out" flickers for ~200ms then reappears because optimistic value only holds DURING the pending transition; RSC `initial` prop still contains the row. Fix: call `router.refresh()` in the `onSuccess` callback (canonical), OR promote optimistic delta to real `useState`. Never assume optimistic state persists post-mutation.
- **Layout-affecting DOM mutations need `useLayoutEffect` (or isomorphic variant), not `useEffect`, for true CLS=0.** Passive `useEffect` runs AFTER paint, so any attribute toggle that drives `body` padding / reserved space causes a one-frame layout shift. SSR-safe pattern: `const useIso = typeof window !== 'undefined' ? useLayoutEffect : useEffect`. Tests asserting "attribute set after render" don't catch this — assert the effect lands in the layout phase via a `vi.doMock('react', ...)` phase-tagged spy. (source: superpowers-exec-tomi · project: kalori)
- **One-shot browser events (`beforeinstallprompt`, `appinstalled`, `pagehide`) MUST register in an always-mounted component — NEVER inside `next/dynamic({ssr:false})` lazy chunks.** The chunk loads asynchronously; any event fired before the chunk resolves is permanently lost (browser does not re-fire). Pattern: always-mount a thin host that owns the listener, pass captured state via PROPS to a lazy modal. Listener registration is non-negotiable; visual rendering can stay lazy. Applies to any framework's lazy-loading primitive. (source: superpowers-exec-tomi · project: kalori)
- **iPadOS desktop-mode UA detection needs three signals AND-ed: `Macintosh` UA + `navigator.maxTouchPoints > 1` + Safari-engine (reject `Chrome|Chromium|CriOS|FxiOS|Edg|OPR`).** Naive `iPhone|iPad|iPod` regex misses iPadOS-in-desktop-mode (default since iOS 13). The Safari-engine constraint rejects Chromium UA-spoofing on touch laptops. Add a unit test with a Macintosh iPadOS UA and a positive control with desktop Mac Safari (maxTouchPoints=0). (source: superpowers-exec-tomi · project: kalori)

## Domain-Specific Notes — Framer Motion

- **[2026-05-08] Framer Motion's `useReducedMotion` only reads the OS `prefers-reduced-motion` signal — apps with their own user-controlled reduce-motion toggles (localStorage flag / `data-attribute` on root / context provider) need a wrapper that ORs all signals; otherwise CSS `@keyframes` animations and Framer animations drift in their honoring of the toggle.** Symptom: user enables in-app "reduce motion" toggle → CSS-animated elements (gated on a `data-attr` matched by `[data-reduce-motion="true"] *` selector) freeze correctly; Framer-animated elements (gated on `useReducedMotion()` from `framer-motion`) keep playing because the framer hook only reads the OS media query. Pattern: wrap once in `lib/motion/defaults.ts` exporting an OR'd hook (e.g. `useReducedMotionApp()` returning `osPref || appPref`); subscribe via `MutationObserver` on `<html>`/`<body>` data-attribute + `StorageEvent` for cross-tab + same-tab `CustomEvent('reduce-motion-changed')` for reactivity within the same window. Consumers (LogFlowModal, WizardShell, MobileWheel*) flow through the wrapper; raw `useReducedMotion` is reserved for code paths that genuinely only care about OS preference (e.g. accessibility audit utilities). Two-baseline-test exception: helpers like `useReducedMotionVariants` invoked outside a component body can keep the raw hook and accept the OS-only limitation as a documented minor — see Phase 7's pending-minor-finding ledger. (source: bugfix-tomi · project: kalori)
- **[2026-05-08] Framer Motion's `animate={{ y: 0 }}` writes `transform: translateY(...)` directly, OVERWRITING any CSS-baked `transform: translate(-50%, -50%)` centering on the same element.** Symptom: a centered modal / overlay element styled with CSS `position: fixed; top: 50%; left: 50%; transform: translate(-50%, -50%)` jumps to the top-left corner the instant Framer's `animate` prop applies because the inline style takes precedence. Pattern fix: SEPARATE the centering element (CSS owner) from the animator element (Framer owner) — wrap `<m.div animate={...}>` in a static centering parent that owns the CSS `transform: translate(-50%, -50%)`, OR animate via a CSS variable that composes with translate (e.g. `--y` CSS var → `transform: translate(-50%, calc(-50% + var(--y)))` — Framer animates the var, CSS retains the centering). The wrapper-parent option is simpler; the CSS-var option keeps the animator on the centered element if the structure must stay flat. Discovery cost: silent on hover/idle, surfaces only when the animation runs; visual baseline tests catch it only if the captured frame is mid-animation. (source: bugfix-tomi · project: kalori)

## Domain-Specific Notes — CSS / Responsive Layout

- **[2026-05-08] `min-width: 0` cascade pattern is the canonical fix for "min-content overflows" in flex/grid containers; without it, a flex/grid child has implicit `min-width: auto` and refuses to shrink below its content's intrinsic width.** Required at EVERY level of the container chain — fixing only the top-level container leaves descendant overflows. Symptom set: at narrow viewports (mobile-375, tablet-768), `document.scrollWidth` exceeds `window.innerWidth` even though the outermost container has `max-width` clamps; child grids / flex rows that contain long text or wide content (charts, tables, code blocks) push the parent's intrinsic width past the clamp because their `min-width: auto` floors at content-intrinsic-width. Pattern: apply `min-width: 0` to (a) the flex/grid container itself, (b) every flex/grid descendant in the chain that holds shrinkable content, (c) any `display: grid` cell with a `1fr` track that holds text content. Pair with an objective overflow regression test — `expect(scrollWidth).toBeLessThanOrEqual(innerWidth)` at every breakpoint covered (mobile-375, mobile-414, tablet-768) — because pixel-diff visual baselines won't catch sub-viewport overflows that scroll horizontally but render visually-correct on capture. Bug #1 + Phase 7 regression of the mobile-ui-overhaul batch: REG-1 `/progress` mobile-375 (151px overflow), REG-2 `/dashboard` tablet-768 (124px), REG-3 `/progress` tablet-768 (30px) — all fixed by adding `min-width: 0` to descendants Bug #1's first pass missed. (source: bugfix-tomi · project: kalori)
