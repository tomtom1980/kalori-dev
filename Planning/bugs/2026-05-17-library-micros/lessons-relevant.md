- **[2026-05-08] False-green tests that assert PRESENCE (e.g., `getByTestId('trigger')` exists) without asserting BEHAVIOR (the trigger does its job) are technical debt with a green checkmark — Codex's adversarial framing "does this primitive serve its stated purpose?" catches them where TDD didn't.** TDD enforces "test must fail then pass," but the gap is in WHAT the test asserts: a test that mounts a component, asks for an element by testid, and verifies non-null can pass while the element does nothing user-observable when activated. C2 in the mobile-ui-overhaul batch shipped under TDD-green tests because the integration test only verified trigger render, not trigger behavior — Codex's R1 framing forced re-asking "what does this primitive PROMISE the user, and does the test exercise that promise?" Pattern at PR review: for every newly-added test that uses `getByTestId` / `getByRole` / `findBy*` without a follow-up `click()` + outcome assertion (state change, network call, focus shift, aria-live announcement), require either (a) the follow-up assertion or (b) a comment explaining why presence alone is sufficient (e.g. snapshot test, smoke test). Distinct from line 100's "live wire, dead consumer" pattern (boundary-write tests passing while consumer ignores the wire) — this is one layer earlier: the test never even probes the wire's behavior. (source: bugfix-tomi · project: kalori)

- **[2026-05-15] Kalori canonical micro-key resolution uses a closed allowlist alias map `{code}_{unit} → code` auto-generated from `DEFAULT_MICROS_LIST` (`sodium_mg → sodium`) — cross-unit suffixes (`sodium_g`) drop silently rather than coerce.** `lib/dashboard/micros-rda-resolver.ts::canonicalizeMicroKey(rawKey)` is the SINGLE source-of-truth helper: resolution order alias → canonical → display-name → undefined. `aggregateMicros` calls it BEFORE the existing display-name translation. Why closed-allowlist + no implicit unit coercion: a `sodium_g` value (grams) silently auto-coerced to canonical `sodium` (mg) would 1000x the RDA contribution and produce wildly-wrong calorie/RDA-pct rollups. Drop-silently is the fail-safe — incorrect data is flagged by absence (RDA pct=0% jumps out), incorrect unit is invisible. Pattern: any code↔legacy key translation map for nutrition / measurement domains uses closed-allowlist semantics; unit-mismatch surfacing is intentional. Applies to future Kalori translation seams (macros, water units mL↔L, weight kg↔lb persistence). Pairs with line 82 (Core Principle: symmetric encoding/translation boundary). (source: superpowers-exec-tomi · project: kalori-mvp-stabilization)

- **[2026-05-14] Encoding/translation boundary fixes are symmetric — when fixing canonical→display drift on one side, audit display→canonical on the OTHER side in the same round.** Kalori C.1 Codex R1 caught canonical AI keys (`vitamin_c`) leaking into the existing `MicronutrientPanel` aggregator which keyed on display names (`Vitamin C`) — fix added `CANONICAL_CODE_TO_DISPLAY_NAME` lookup in `aggregate.ts`. Round 2 caught the SYMMETRIC gap: the new `resolveMicrosRda` resolver only read `micros[canonical]` and silently dropped legacy display-name keys from persisted/cached rows — fix added `DISPLAY_NAME_TO_CANONICAL_CODE` (inverse map, single source of truth in `lib/nutrition/micros-rda.ts`) and a key-resolution loop honoring both shapes. Discipline: when an encoding boundary is identified (canonical↔display, codepoint↔glyph, internal-unit↔external-unit, snake_case↔camelCase), fix BOTH directions in the same round and ship the inverse map alongside the forward map. Two pointers, two lookups, one source of truth. Pairs with line 146 (sibling path symmetry: grep all callsites) — that's about producer-side symmetry; this is about producer-vs-consumer symmetry across a translation seam. Applies to any encode/decode pair (i18n key↔display, normalized↔raw, internal↔wire). (source: superpowers-exec-tomi · project: kalori-mvp-stabilization)

- **Unit conversion MUST precede validation, not follow it.** A numeric input accepted in `lb` that passes a range check (e.g. 150 lb within 30–350) and POSTs as `kg` silently corrupts at 2.2×. Convert at the component boundary BEFORE Zod/range validation so the validated value IS the canonical internal unit. Never let external unit semantics leak past the validation layer — no error fires, the data is just wrong.

- **[2026-05-17] Mocking the HTTP client bypasses the shared validation schema — green tests can mask a production 400.** When the prod route runs request bodies through a shared Zod (or equivalent) schema (e.g. `CreateLibraryBodySchema`) but unit tests mock the client (`authFetch`, `fetch`, axios) at the call boundary, the asserted body shape never traverses the validator. A contract violation (UUID-shaped field receiving `${clientId}:${idx}` suffix that's no longer a UUID) ships green. Defense-in-depth: pair client-mock tests with EITHER (a) a unit test that runs a representative emitted body through the same shared schema (`expect(CreateLibraryBodySchema.safeParse(body).success).toBe(true)`) OR (b) a thin integration test against the actual route handler. Distinct from line 41 (dead-helper tests) and line 140 (mock-writes-always-succeed): this is one layer higher — the test never exercises the shared validator that the production path enforces. (source: superpowers-exec-tomi · project: kalori-mvp-stabilization)

- **[2026-05-15] Two-reviewer Phase 3 catches what one misses — the SECOND independent reviewer covers a complementary axis (design/a11y vs perf/correctness), and the two finding sets do not overlap.** Kalori C.2's Phase 3 dispatched parallel `design-a11y` + `react-perf` sub-agents: design-a11y caught undefined CSS classes (`kalori-re-*` referenced across new Recent Entries surface but never declared in `globals.css`) + missing in-flight Log-Now cue (no disabled / aria-busy / label swap) + missing alertdialog role on N=1 BulkDeleteConfirm; react-perf independently caught a `ate_at` vs `logged_at` field-name mismatch between client POST body and server `.strict()` Zod schema (HTTP 400 in production despite passing tests) + dead `<Suspense>` boundary wrapping a non-async `Promise.all`-resolved RSC + redundant `revalidatePath` on the log-now route. A single reviewer asked to "cover both axes" routinely misses one — the cross-axis bundling produces a single shallow finding set. Pattern: at Phase 3 of any UI-bearing task, dispatch ≥2 reviewers with disjoint axis prompts (a11y/design-system vs perf/correctness; or a11y vs security; or design-system vs cross-task brownfield impact). Each reviewer reads only their axis; orchestrator dedupes only if both flag the same defect. Cost is ~2× sub-agent fan-out; value is catching defects the bundled-prompt reviewer never would. Pairs with line 199 (Phase 3 structurally verifies Click-Through Mandate) — same Phase 3 surface, different value-add (this is reviewer multiplicity; that is push-upstream-from-C9). Applies to any review checkpoint where multiple independent quality axes exist. (source: superpowers-exec-tomi · project: kalori-mvp-stabilization)

- **[2026-04-30] Tests must verify EFFECTIVE behavior, not boundary writes.** A toggle that writes localStorage + sets a `data-attr` can pass behavioral tests while the consumer hook ignores both inputs and the CSS mirror selector is missing. Pattern: integration test must read computed style / actual rendered output AFTER the toggle, not just confirm the boundary side-effects fired. Why: surface-level mocks/spies are happy when the wire is connected on one end and dangling on the other — only end-to-end effect assertions catch full disconnects. Adjacent to the dead-helper-test pattern but the failure mode is "live wire, dead consumer" not "test on orphan."

- **[2026-05-16] Wave / cluster sub-agents implementing per-bug fixes MUST receive project-wide audit + contract files in their briefing (focus-ring tokens, nav-audit pragmas, schema-drift hash markers, design-system token allowlists), not only the bug-scoped Reads list — otherwise the same project-wide invariant gets violated twice in parallel waves.** Kalori library-overhaul batch: Wave 3 (Bug 10 hover focus) AND Wave 5 (Bug 6 add-item dialog) both introduced `:focus-visible { outline-color: var(--color-oxblood-soft) }` declarations DESPITE the lessons file already carrying line 419 ("`--color-oxblood-soft` is ACCENT-ONLY on dark surfaces — failing WCAG AA"). Each wave's bug-scoped TDD was GREEN because the wave's own unit tests didn't run the project-wide focus-ring-token integration test; only the full sweep caught two simultaneous violations of the same canonical rule. Pattern at briefing-time: the wave sub-agent prompt enumerates BOTH (a) the bug-specific files, AND (b) project-wide invariant contracts the sub-agent will land code under — for Kalori that's `tests/integration/focus-ring-token.test.ts`, `tests/integration/nav-audit.test.ts`, `scripts/schema-drift-check.mjs`, `Planning/ui-design.md` token-allowlist sections. The sub-agent's GREEN gate must include "run the project-wide invariant tests once your scoped tests pass" before reporting completion. Cost: a few seconds of test runtime per wave; savings: zero post-wave audit-fix sub-agents and a tighter Codex round 1. Pairs with line 211 (sub-agent brief "optionally add X" → author owns auditing ALL X sites) and line 235 (each Codex fix-layer needs its own threat model) but applies one phase earlier: at WAVE briefing-time, before the sub-agent has done any work. Applies to any parallel-wave fix orchestration where project-wide design-system / a11y / contract invariants exist. (source: bugfix-tomi · project: kalori)
