## Core Principles — Review Discipline

- **[2026-05-08] `startTransition` does NOT commit `setBusy(true)` synchronously — same-tick double-submit bypasses the React-state guard.** A submit handler that reads `if (busy) return` at the top, then calls `setBusy(true)` inside `startTransition(() => ...)`, ships a re-entrancy hole: two synchronous `requestSubmit()` / Enter-key fires in the same tick both observe `busy === false` and both reach the network. Pattern: `useRef<boolean>` latch checked AND set BEFORE entering any transition (`if (inFlightRef.current) return; inFlightRef.current = true; startTransition(...)`); cleared in finally + on unmount. React state is for renders, refs are for synchronous gates. Applies to ANY form using `startTransition` + an "in-flight" guard, including server-action forms. (source: superpowers-exec-tomi · project: kalori-mvp-stabilization)

- **[2026-05-08] `setTimeout` scheduled in an async success branch fires on a destroyed component unless guarded at BOTH schedule site AND callback.** Unmount cleanup that clears the *currently-pending* timer ID does NOT prevent a NEW timer being scheduled later when `await fetch()` resolves AFTER the user navigates away. The defer pattern (`onSuccess → setTimeout(() => router.refresh(), 200)`) lands `router.refresh()` on whatever route the user is on now. Pattern: `mountedRef` checked before scheduling the timer AND inside the timer callback; `useEffect` cleanup sets `mountedRef.current = false`. Same trap with `requestAnimationFrame`, `queueMicrotask`, and any post-await side effect. (source: superpowers-exec-tomi · project: kalori-mvp-stabilization)

- **[2026-05-08] When N consumers call the same helper and a contract changes, fix at the helper, not at every consumer — surgical AND surface-shrinking.** Fence-style preflights (orphan-profile fence, auth gate, feature-flag check) often live in a shared helper invoked from N pages/route handlers. When the helper's contract changes (e.g. `throw 401` → `redirect('/onboarding')`), the temptation is to update each consumer's error-handling block. Discipline: prefer one fix at the helper that mutates behavior at the source AND shrinks the surface (one branch instead of N), unless a per-consumer reason exists (different fallback UI, different telemetry tag). Why: per-consumer fixes drift over time as new consumers ship without remembering to update; helper-level fix is automatic for any future consumer. Distinct from "Sibling path symmetry", which says fix-all-siblings-in-same-commit when the duplication can't be DRYed; this rule says DRY first, fix the surface, when the helper exists. Applies to any duplicated pattern across N callsites where a single helper would naturally own the contract. (source: superpowers-exec-tomi · project: kalori-mvp-stabilization)

- **[2026-05-08] False-green tests that assert PRESENCE (e.g., `getByTestId('trigger')` exists) without asserting BEHAVIOR (the trigger does its job) are technical debt with a green checkmark — Codex's adversarial framing "does this primitive serve its stated purpose?" catches them where TDD didn't.** TDD enforces "test must fail then pass," but the gap is in WHAT the test asserts: a test that mounts a component, asks for an element by testid, and verifies non-null can pass while the element does nothing user-observable when activated. C2 in the mobile-ui-overhaul batch shipped under TDD-green tests because the integration test only verified trigger render, not trigger behavior — Codex's R1 framing forced re-asking "what does this primitive PROMISE the user, and does the test exercise that promise?" Pattern at PR review: for every newly-added test that uses `getByTestId` / `getByRole` / `findBy*` without a follow-up `click()` + outcome assertion (state change, network call, focus shift, aria-live announcement), require either (a) the follow-up assertion or (b) a comment explaining why presence alone is sufficient (e.g. snapshot test, smoke test). Distinct from "live wire, dead consumer" pattern (boundary-write tests passing while consumer ignores the wire) — this is one layer earlier: the test never even probes the wire's behavior. (source: bugfix-tomi · project: kalori)

- **Tests asserting on a dead helper still pass green.** When refactoring or removing infrastructure, grep test files for the removed helper name and confirm assertions hit the LIVE pipeline (e.g. real route table), not the orphan function. Codex caught this on Kalori 5.1.2 even with 36 PWA tests green. (source: superpowers-exec-tomi · project: kalori)

- **Codex catches read-compute-write races and error-swallow patterns TDD misses.** Prefer payload-only compute + atomic `upsert`/`update` over SELECT→merge→UPDATE. Always distinguish the `error` branch from `data == null` at every `.maybeSingle()` callsite.

- **Fixing request SHAPE is not fixing ERROR BEHAVIOR.** When a Codex Round 1 fix corrects the POST body schema, the next-round finding is often the unchanged `try/finally` that swallows server errors and hides the UI on `finally`. UI announces success after server 4xx/5xx. Test the failure branch explicitly (mock 500, assert toast stays, assert card stays visible) in the same commit as the shape fix — Round 2 will otherwise find it as a regression, not a new defect.

## Core Principles — Testing

- **[2026-04-30] Tests must verify EFFECTIVE behavior, not boundary writes.** A toggle that writes localStorage + sets a `data-attr` can pass behavioral tests while the consumer hook ignores both inputs and the CSS mirror selector is missing. Pattern: integration test must read computed style / actual rendered output AFTER the toggle, not just confirm the boundary side-effects fired. Why: surface-level mocks/spies are happy when the wire is connected on one end and dangling on the other — only end-to-end effect assertions catch full disconnects. Adjacent to the dead-helper-test pattern but the failure mode is "live wire, dead consumer" not "test on orphan."

- **[2026-05-08] E2E `waitForTimeout(N)` after a fire-and-forget POST is always wrong — replace with `waitForResponse(predicate)` registered BEFORE the action.** Fixed-duration sleeps after a POST that round-trips through a cross-region pair (e.g. Vercel `iad1` ↔ Supabase `ap-southeast-1` ≈ 150–200ms one-way RTT, plus server processing) flake under contention even when the value (e.g. 500ms) is "comfortably" larger than median latency. The deterministic pattern: `const undoResponse = page.waitForResponse(r => r.url().includes('/api/...') && r.request().method() === 'POST' && r.status() === 200); await page.click(...); await undoResponse;` — registers the listener BEFORE the click so the response can't fire before `waitForResponse` is attached, and asserts the actual contract (URL match + method + status) rather than a wall-clock proxy. Same pattern beats `waitForTimeout` in any scenario where the test's logical "next step" is "wait for this server side-effect to land" — undo POSTs, optimistic-update reconciliations, mutation-then-RSC-revalidation flows. The cost is one extra line; the savings are zero flakes under cross-region latency, parallel-worker contention, or CI cold-start. (source: bugfix-tomi · project: kalori)

- **[2026-05-07] When a server-side cache-invalidation primitive (`revalidatePath`) coexists with a client-side self-hydration codepath, E2E must PROVE the server primitive is load-bearing — not be silently propped up by client polling.** Pattern: attach `page.route('/api/<self-hydrate-route>', route => { calls.push(...); route.continue() })` listener BEFORE the user action. After the action + UI assertion lands, assert `calls.length === 0` (post-action UI state must come from the FIRST RSC payload, not a subsequent client fetch). If the listener fires non-zero, emit a console.warn NOTABLE — the server invalidation may be dead code masked by client redundancy. Applies to any "server `revalidatePath`/`revalidateTag` + client SWR/polling/tRPC subscription" combo where regression in the server primitive would silently degrade to client-only behavior. (source: superpowers-exec-tomi · project: kalori-mvp-stabilization)

- **[2026-05-01] Test scaffolds that mock writes to ALWAYS succeed (`error: null`) make the error-path invisible to TDD.** A `buildMocks()` helper that hard-codes `food_library_items.insert(...).single()` to `{ data: row, error: null }` keeps 1735/1735 GREEN while production fires `revalidateTag` + `revalidatePath` on PostgREST 23505 / RLS denial / schema drift. Codex's adversarial pass reads error-path branches that mocked TDD doesn't exercise. Pattern: when adding a test for any write that uses a shared mock-builder, ALSO add a sibling test that overrides the mock to return `{ data: null, error: { code, message } }` and asserts side-effects (revalidate, audit, downstream writes) DID NOT fire. Pairs with the existing PostgREST `.maybeSingle()` lesson but applies at TEST scaffold level, not callsite level. (source: superpowers-exec-tomi · project: kalori)

## Core Principles — Error Handling

- **[2026-05-09] Server-side error fallbacks must fail loud — `try { ... } catch { return DEFAULT }` masks real defects; capture to Sentry (or equivalent) BEFORE the fallback returns or you'll never know the catch fired.** Codex R1 Critical C1 in the mobile-water-button batch caught a `(app)/layout.tsx` profile-timezone fetch where a column-name typo (`user_id` instead of `id`) made every Supabase query throw `PGRST116`-class errors, but the `catch` block silently returned `'UTC'` so non-UTC users had `loggedOn` derived from server-side UTC midnight — wrong-day writes in production with no signal. Fix added `Sentry.captureException(error, { tags: { component: 'layout', concern: 'timezone-derivation' } })` BEFORE the UTC fallback so the silent path becomes visibly noisy when it fires. Pattern: any catch block that returns a default value (timezone, locale, feature-flag default, "best guess" value) needs an explicit log/capture/metric at the catch site — even when the fallback itself is "graceful." Distinguish from `catch { swallow }` after a state commit (that's about state-machine consistency); this rule applies to read-side catch-and-default where silent failure produces wrong-but-rendered output. The fallback may still ship; it just must not be silent. Applies to RSC server fetches with default values, client-side `try { JSON.parse(...) } catch { return {} }`, any catch returning a "safe" default. (source: bugfix-tomi · project: kalori)

## Core Principles — Process & Sub-agents

- **[2026-05-09] Drill source values, not derived values — when a prop depends on "now" (today's date, current locale, current timezone), drill the INPUTS and recompute at use-time; precomputing in the parent and prop-passing captures the value at render time and goes stale across long-lived sessions.** Codex R2 Critical C2 in the mobile-water-button batch caught a `loggedOn` prop captured at server-render in `(app)/layout.tsx` and threaded through `<NavShell>` → after a session crossed local midnight, the FAB still POSTed yesterday's date. Fix drilled `timezone` (the input) instead of `loggedOn` (the derived value), and recomputed `userTzToday(timezone)` synchronously inside the click handler at tap time. Pattern: if `value = compute(input, now())`, drill `input` and call `compute(input, now())` at the moment of use; never drill `value`. Applies to any "today's date" / "current locale" / "active feature flag" / "current viewport class" prop on a long-lived component. The render-time-vs-use-time gap widens with session duration — short-session UIs may never observe it; long-lived dashboards / always-mounted nav chrome / PWAs in standalone mode reliably cross the gap. Pairs with `setTimeout`-after-await mountedRef pattern (also a render-time-vs-use-time gap) but applies to read-side derived values rather than write-side timer scheduling. (source: bugfix-tomi · project: kalori)

## Domain-Specific Notes — Next.js 16 (App Router + Turbopack)

- **React 19 `useOptimistic` state decays the instant the action resolves.** Card "fade out" flickers for ~200ms then reappears because optimistic value only holds DURING the pending transition; RSC `initial` prop still contains the row. Fix: call `router.refresh()` in the `onSuccess` callback (canonical), OR promote optimistic delta to real `useState`. Never assume optimistic state persists post-mutation. **[FLAG: directly relevant to Bug 2]**

- **`useTransition().isPending` does NOT track Promise lifetime — only React rendering priority.** Wrapping `startTransition(async () => await sideEffect())` schedules the callback then drops `isPending` BEFORE the awaited Promise resolves. For UI that must reflect "request in flight" (disable retry buttons, show spinner), use explicit `useState<boolean>` set true before `await`, cleared in `finally`. Keep `useTransition` only for rendering-priority hints. (source: superpowers-exec-tomi · project: kalori)

- **Auth-gated RSCs MUST `export const dynamic = 'force-dynamic'`** — or they fail static prerender in CI builds without Supabase envs. Add a text-level regression guard test (grep the page source for the export). Prefer adding at the LAYOUT level when the layout becomes async-with-Supabase-auth — one export covers all child routes.

- **`unstable_cache(fn, keyParts, { tags })` cannot see `cookies()` — the closure runs outside request scope.** Two fixes: (a) hoist auth OUT of the cached function (resolve `userId` in request scope FIRST, pass as keyParts); (b) use `React.cache()` for per-request dedupe only.

- **[2026-05-01] Under `cacheComponents: false`, `revalidateTag` / `cacheTag` / `'use cache'` are forward-compat no-ops — `revalidatePath('/route', 'page')` is the load-bearing primitive for Router-Cache invalidation from a Route Handler.** `'use cache'` requires `cacheComponents: true` to function (build-fails otherwise); `cacheTag(...)` calls inside `unstable_cache` wrappers are silently ignored when no consumer subscribes via `cacheTag`. Keep the `revalidateTag` calls in mutation handlers as forward-compat for the eventual `cacheComponents:true` flip, but ADD `revalidatePath('/library', 'page')` (or equivalent) for the actual cache-invalidation contract under the current regime. Pairs with the `unstable_cache + cookies()` lesson: both require knowing the project's runtime cache mode before authoring fixes. **[FLAG: directly relevant to Bug 2 — `router.refresh()` may be insufficient if `revalidatePath` is missing on the route handler]**

- **React 19 RSCs pass ONLY serializable props to client components — no function children / render-props / callback props across server→client boundary.** Canonical fix: client leaf owns all conditional logic; server passes serializable arrays + counts.

## Domain-Specific Notes — Kalori (AI calorie tracker, Next 16 / Supabase / Gemini)

- **F12 R1 mitigation contract has held across Phase 2.** Centralized `lib/auth/refresh-interceptor.ts` (`authPost` / `authFetch` / `SessionExpiredError`) is mandatory ingress for any auth-sensitive mutation; Phase 3/4 inherit without local refresh shims. Keep the forbidden-local-shim rule load-bearing.

- **Single-item soft-delete reuses the bulk substrate** — `/api/library/bulk-delete/undo` + `useUndoQueueStore.pushToast` + `<UndoToast>` accept length-1 `client_ids` arrays; never scaffold a `/api/library/[id]/undo` route. `kalori-library-pill` class (`app/globals.css:1981`, 8.86:1 AAA on bg-0) is the canonical destructive/primary CTA — reuse verbatim, don't re-derive per component.

- **[2026-05-09] Kalori `profiles` table column for FK-mirroring `auth.users.id` is `id`, NOT `user_id` — copy from an existing call site rather than guessing.** Server-component layout fetches and any RSC-side `profiles` lookup must use `.eq('id', user.id)`; the column name `user_id` (which appears on other Kalori tables like `food_log_entries`, `weight_log`, `water_log`) does NOT exist on `profiles`. Pattern: when adding a new `profiles`-table call site, grep an existing one (`app/(app)/dashboard/page.tsx` is a known-good example) and copy the column shape verbatim. Same pattern applies to other FK-mirroring tables (the Kalori convention is `id` when the row IS the user, `user_id` when the row REFERENCES the user). (source: bugfix-tomi · project: kalori)
