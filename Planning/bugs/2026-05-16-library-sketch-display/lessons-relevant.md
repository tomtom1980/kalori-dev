Pattern applies to any Supabase Storage bucket with path-based RLS where the asset URL must be presentable: store path, sign on read, set TTL aligned with the longest plausible page-view session (1 hour fits Kalori; longer TTLs leak into security review). - **[2026-05-16] `sharp.failOn` ladder is `'none' < 'truncated' < 'error' < 'warning'` — the DEFAULT is `'warning'` (strictest), so explicit `failOn: 'truncated'` is LOOSER than not setting it.** The option-name reads "stricter" than the default to most readers (truncated sounds like a sterner gate than warning) but the docs lattice is the reverse: `'none'` accepts everything, `'warning'` rejects on the first warning. Mini-batch A SEC-M1 fix set `failOn: 'truncated'` thinking it tightened decode validation against malformed PNGs; Codex caught that it weakened the default. Correct pattern: omit the option (use default `'warning'`) OR explicitly write `failOn: 'warning'`. Same trap class applies to any sharp option whose name sounds stricter than the docs ladder reveals — always cross-reference the lattice rather than relying on intuitive naming. Adjacent gotchas in the same surface: `sequentialRead: true` (memory hint, not strictness), `limitInputPixels: false` (DISABLES the input-pixel cap; explicit numeric ceiling required for hostile-input protection). (source: bugfix-tomi · project: kalori)
