- **[2026-05-16] iOS-only WebKit behaviors that require an OS-rendered overlay (native date picker, native time wheel, file-picker chrome, share-sheet) CANNOT be functionally verified via Playwright webkit-on-desktop emulation — pair jsdom structural-invariant unit tests (`getComputedStyle().pointerEvents`, `elementFromPoint` substitutes, wrapper min-width/min-height numbers) with Playwright webkit-engine structural assertions (`document.elementFromPoint(cx, cy)`, `getBoundingClientRect()` ≥ 44×44, `pointer-events: none` on decorative overlays) — the union covers the "tap target is the right DOM node" contract without a real device farm.** The native picker overlay only exists on actual iOS Safari; webkit-on-desktop reproduces hit-testing geometry, JIT, layout, and the rendering pipeline but NOT the OS-level UI chrome that opens on tap. Verification has to be STRUCTURAL: prove the right element receives the tap (via `elementFromPoint` at the centre coordinate of the tap target), prove decorative overlays carry `pointer-events: none` (cannot steal taps), prove minimum size meets WCAG 2.5.5 + Apple HIG 44pt (≥ 44×44), prove accessible label is on the operable input not a sibling. Pattern: at unit layer assert `getComputedStyle(input).pointerEvents !== 'none'` + wrapper `minWidth/minHeight ≥ 44` numbers + ARIA-label resolves to the input element itself; at E2E layer add a webkit-iphone + webkit-ipad Playwright project pinned to `devices['iPhone 15 Pro']` / `devices['iPad Pro 11']` and use `evaluate()` to call `document.elementFromPoint(rect.x + rect.width/2, rect.y + rect.height/2)` and assert the returned element matches the operable input by attribute/testid. Real-device verification (BrowserStack, physical iPhone) remains the only ground truth for picker chrome itself; structural coverage is what the agent can ship. Distinct from line 147 (CI-DEFERRED classification rule — that's about WHERE the test runs); this is about WHAT to assert when the OS chrome is unreachable. (source: bugfix-tomi · project: kalori-mvp)

- **[2026-05-15] Kalori `--color-oxblood-soft` (#a13a2c) is ACCENT-ONLY on dark surfaces — text use against `--color-bg-1` (#0E0A08) yields 2.96:1, against `--color-bg-quote` (#1a1310) yields 2.75:1; both fail WCAG AA normal-text 4.5:1.** Allowed uses on dark backgrounds: border-color, glyph-on-tile, focus-outline, hoverBg accent. Forbidden uses: body text, button labels, kicker spans, any rendered text >= 1px font-size that axe will scan. Forbidden sites resolved in D.E2E: `components/dashboard/MealEntryContextTrigger.tsx` (meal-add labels, now `--color-ivory`), `components/charts/WeeklyReviewCore.tsx` (sparse-state kicker via parent `<p>` color cascade, now `--color-ivory`). Safe text use on ivory surfaces: `app/(auth)/login/...`, `app/(app)/onboarding/...` — all `--color-bg-ivory` backgrounds clear AA with oxblood-soft text. Aria-hidden text uses (`aria-hidden="true"` on dash glyphs in `MicrosOverflowToggle.tsx`, full-state `WeeklyReviewCore.tsx:109` kicker variant, `AccountDeleteFlow.tsx` glyph) are NOT flagged by axe color-contrast and stay as-is. Pattern: when a Kalori component needs an accent + a text-label on the same dark surface, use oxblood-soft for the accent (border, glyph, focus-ring) and ivory for the text. Pairs with line 391 (Inverse-pill is the WCAG-safe CTA pattern for the SOLID-oxblood case) — that's about CTAs; this is about ambient accent text on dark surfaces. (source: superpowers-exec-tomi · project: kalori-mvp-stabilization)

- **Inverse-pill is the WCAG-safe CTA for dark-themed brands with a dominant non-passing brand color.** Dark bg (near-black) + dominant brand color (oxblood, crimson, dark-blue) often fails SC 1.4.3 (2.28:1 at `#8A2A1F` on `#0E0A08`). Flipping to brand-color-filled pill + light-text-on-dark-pill inverts the contrast (8.86:1 AAA) while preserving brand identity. Applies to primary + destructive CTAs. Document as WCAG tiebreaker over visual spec.

Cost: a few seconds of test runtime per wave; savings: zero post-wave audit-fix sub-agents and a tighter Codex round 1. Pairs with line 211 (sub-agent brief "optionally add X" → author owns auditing ALL X sites) and line 235 (each Codex fix-layer needs its own threat model) but applies one phase earlier: at WAVE briefing-time, before the sub-agent has done any work. - **[2026-05-08] `min-width: 0` cascade pattern is the canonical fix for "min-content overflows" in flex/grid containers; without it, a flex/grid child has implicit `min-width: auto` and refuses to shrink below its content's intrinsic width.** Required at EVERY level of the container chain — fixing only the top-level container leaves descendant overflows. Symptom set: at narrow viewports (mobile-375, tablet-768), `document.scrollWidth` exceeds `window.innerWidth` even though the outermost container has `max-width` clamps; child grids / flex rows that contain long text or wide content (charts, tables, code blocks) push the parent's intrinsic width past the clamp because their `min-width: auto` floors at content-intrinsic-width. Pattern: apply `min-width: 0` to (a) the flex/grid container itself, (b) every flex/grid descendant in the chain that holds shrinkable content, (c) any `display: grid` cell with a `1fr` track that holds text content. Pair with an objective overflow regression test — `expect(scrollWidth).toBeLessThanOrEqual(innerWidth)` at every breakpoint covered (mobile-375, mobile-414, tablet-768) — because pixel-diff visual baselines won't catch sub-viewport overflows that scroll horizontally but render visually-correct on capture. Bug #1 + Phase 7 regression of the mobile-ui-overhaul batch: REG-1 `/progress` mobile-375 (151px overflow), REG-2 `/dashboard` tablet-768 (124px), REG-3 `/progress` tablet-768 (30px) — all fixed by adding `min-width: 0` to descendants Bug #1's first pass missed. (source: bugfix-tomi · project: kalori)

- **[2026-05-08] False-green tests that assert PRESENCE (e.g., `getByTestId('trigger')` exists) without asserting BEHAVIOR (the trigger does its job) are technical debt with a green checkmark — Codex's adversarial framing "does this primitive serve its stated purpose?" catches them where TDD didn't.** TDD enforces "test must fail then pass," but the gap is in WHAT the test asserts: a test that mounts a component, asks for an element by testid, and verifies non-null can pass while the element does nothing user-observable when activated. C2 in the mobile-ui-overhaul batch shipped under TDD-green tests because the integration test only verified trigger render, not trigger behavior — Codex's R1 framing forced re-asking "what does this primitive PROMISE the user, and does the test exercise that promise?" Pattern at PR review: for every newly-added test that uses `getByTestId` / `getByRole` / `findBy*` without a follow-up `click()` + outcome assertion (state change, network call, focus shift, aria-live announcement), require either (a) the follow-up assertion or (b) a comment explaining why presence alone is sufficient (e.g. snapshot test, smoke test). Distinct from line 100's "live wire, dead consumer" pattern (boundary-write tests passing while consumer ignores the wire) — this is one layer earlier: the test never even probes the wire's behavior. (source: bugfix-tomi · project: kalori)

- **[2026-04-30] Contrast / a11y assertions on TOKEN NAMES are not enforcement.** Regex matching `var(--color-ivory)` literals proves the swap landed, NOT that the resolved color clears the threshold. Resolve `var(--token)` to `#hex` via a test-side token map, parse to sRGB tuple, compute WCAG ratio numerically, assert against the AAA/AA constant. Helper lives test-only — zero new runtime deps. Why: regex tests went green while real contrast was 3.26:1 / 2.28:1 (FAIL). Same trap applies to focus-ring color, focus-ring width-via-token, and any visual contract that resolves through CSS custom properties at paint time.

- **[2026-05-14] `prettier-plugin-tailwindcss` can silently strip a leading-space template-literal fragment, restoring a className-concatenation bug a fix had just resolved.** Symptom: writing `` className={`base-class${cond ? ' is-readonly' : ''}`} `` to gate a state modifier produces `base-classis-readonly` (one mashed token) after the pre-commit prettier hook runs — the leading whitespace inside the conditional fragment is collapsed, and CSS selectors `.base-class.is-readonly` never match. The bug-fix commit lands GREEN locally but the prettier hook silently re-introduces the original defect on commit. Prettier-stable pattern: `` className={[..., cond && 'is-readonly'].filter(Boolean).join(' ')} `` — the array-filter-join syntax gives prettier no whitespace ambiguity to collapse. Same trap with `cn()` helper IF the helper hasn't yet been imported and the obvious "just put a space in the template literal" is the reflexive workaround. After every className-concat fix, run the formatter and verify the post-hook output preserves the fix — formatter hooks have opinions, and bug-fixes that depend on leading/trailing whitespace are exposure. Distinct from line 101 (token NAME regex vs computed style) — that's about assertion vacuity; this is about formatter-introduced regression. (source: superpowers-exec-tomi · project: kalori-mvp-stabilization)
