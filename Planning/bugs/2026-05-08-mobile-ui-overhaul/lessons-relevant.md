- **[2026-05-08] E2E `waitForTimeout(N)` after a fire-and-forget POST is always wrong — replace with `waitForResponse(predicate)` registered BEFORE the action.** Fixed-duration sleeps after a POST that round-trips through a cross-region pair (e.g. Vercel `iad1` ↔ Supabase `ap-southeast-1` ≈ 150–200ms one-way RTT, plus server processing) flake under contention even when the value (e.g. 500ms) is "comfortably" larger than median latency. The deterministic pattern: `const undoResponse = page.waitForResponse(r => r.url().includes('/api/...') && r.request().method() === 'POST' && r.status() === 200); await page.click(...); await undoResponse;` — registers the listener BEFORE the click so the response can't fire before `waitForResponse` is attached, and asserts the actual contract (URL match + method + status) rather than a wall-clock proxy. Same pattern beats `waitForTimeout` in any scenario where the test's logical "next step" is "wait for this server side-effect to land" — undo POSTs, optimistic-update reconciliations, mutation-then-RSC-revalidation flows. The cost is one extra line; the savings are zero flakes under cross-region latency, parallel-worker contention, or CI cold-start. Bug 2 + Bug 3 of the e2e-regressions batch were both this exact failure mode — bulk-delete-undo and single-delete-undo flaked on `waitForTimeout(500)` and went 100% green on `waitForResponse(...)` after 30 reps. (source: bugfix-tomi · project: kalori)
- **[2026-05-08] Test fixtures that skip on auth-redirect MUST include a never-skipping canary that probes the OPPOSITE direction.** When a forged-cookie / unauth-fixture E2E suite uses `test.skip(...)` whenever the page lands on `/login` (because the forged session can't reach the protected wizard until F-TEST-4 lands a real test user), the suite's "all green" signal is indistinguishable from the catastrophic regression where every authenticated user gets force-redirected to `/login` — both produce 100% skip-clean. Adversarial review's Round 2 framing: "could a real auth-guard regression skip this suite green?" — the answer is yes unless the suite includes a positive-direction canary that NEVER skips. Pattern: add ONE always-running smoke test that clears all cookies and asserts the unauth-redirect-to-login fence still fires (proves the gate works in the negative direction); pair with an `afterAll` outcome tracker that records skip-vs-pass per test and `console.warn` (or in CI: `throw`) when 100% skipped via the auth-redirect path. The canary protects against the failure mode where the gate is broken in BOTH directions; the outcome tracker protects against silent attrition. Defense in depth until a real authenticated fixture can replace the skip-on-redirect disposition. Applies to any E2E suite that defers the protected-path test until an auth fixture lands. (source: bugfix-tomi · project: kalori)
- **[2026-05-08] `app/(app)/loading.tsx` (Next 16 RSC nav skeleton) resolves Playwright's `page.goto()` BEFORE a server-side `redirect()` commits — synchronous `expect(page.url()).toContain('/login')` flakes; switch to `await page.waitForURL(/<pattern>/)`.** A protected route with both a server `redirect()` (e.g. `redirect('/login?reason=...')` from an auth fence) AND a sibling `loading.tsx` boundary paints the skeleton during the redirect commit window; Playwright sees the navigation as "complete" against the pre-redirect URL while the client-side router is still committing the new URL. Probe-spec evidence: post-fix dashboard `goto` resolves immediately at `/dashboard` (skeleton painted), then within ~ms the URL flips to `/login?reason=session_expired&redirect_to=%2Fdashboard`. Synchronous `page.url()` reads the pre-flip URL and the test fails despite the production redirect contract being correct. Pattern: replace `expect(page.url()).toContain('/login')` with `await page.waitForURL(/\/login\?.*redirect_to=/, { timeout: 5000 })` (matches the actual asserted contract: redirect happened with redirect_to query param). Same trap on any RSC redirect path under a `loading.tsx` boundary — auth fences, role-gated pages, feature-flag bounces. Distinct from `waitForResponse` for fire-and-forget POSTs: this is for navigation/redirect commit timing, not server round-trip timing. (source: bugfix-tomi · project: kalori)
- **[2026-05-08] When changing a contract on shared auth/fence code, audit ALL callers AND all assertions in their tests in the same patch — and verify the change is necessary at all.** B.SWEEP changed orphan-fence's `lookup_error → throw` to `lookup_error → redirect('/onboarding')` to "self-heal a transient missing-row case," but `.maybeSingle()` already routes truly-missing rows through the `orphan` branch (`data:null, error:null`). The contract change was a misdiagnosis: it didn't fix a real bug AND it broke the C1-B forged-cookie regression guard (RLS-denial errors masquerading as orphan → silently land on `/onboarding` instead of `/login`). Only 4 page-guard tests were re-aligned; the forged-cookie E2E was deferred to followups. Discipline: (a) before changing a shared-helper contract, prove the new behavior is needed (grep for the symptom path, write the failing test that the OLD contract can't satisfy); (b) audit every consumer + every assertion across the test tree (`grep -rn "<old-contract-marker>" tests/`) in the SAME patch — not "deferred to followups"; (c) if the symptom is already handled by an adjacent branch (here: `orphan` branch handles missing rows via maybeSingle's null-data path), revert the contract change rather than ship it. The fix that closes the gap is often the revert + a narrow defense-in-depth carveout (here: PGRST116-only redirect inside `lookup_error` for the rare edge case where Supabase returns no-row as an error rather than null data) — preserves the original contract while honoring the original "missing row → onboarding" intent. Pairs with line 134 (Sibling path symmetry) but applies one layer up: before fixing siblings, verify the original contract change was needed at all. (source: bugfix-tomi · project: kalori)
- **[2026-05-08] 2-round fix circuit breaker semantics: fix the SPEC if the bug is in the spec; HALT if the bug is in production code.** During an E2E task, when a Round 1 test failure traces to a real production bug (not a flaky test, not an AC-divergence), DON'T silently patch production behavior to satisfy the spec — that conflates the E2E task's charter (test-authoring) with production refactor scope. Pattern: (a) confirm the production gap via inspection of the implicated source file, (b) refine the test to a smoke-level observable (e.g. coarse-grain assert via toast/server-data round-trip rather than direct predicate assertion) so the AC still passes against current production behavior, (c) emit a `console.warn` flag during the AC run for visibility, (d) file a NEW followup with severity Architectural / Critical naming the production gap, (e) surface the gap in the evidence narrative so reviewers see it. The followup is the right surface for the production fix, scheduled at the right gate (phase Codex review or post-phase polish). Refines the existing 2-round-cap principle: round 1 = "is this our test's defect or production's defect?" — the answer changes the surface that fix lands on. (source: superpowers-exec-tomi · project: kalori-mvp-stabilization)
- **[2026-05-08] When N consumers call the same helper and a contract changes, fix at the helper, not at every consumer — surgical AND surface-shrinking.** Fence-style preflights (orphan-profile fence, auth gate, feature-flag check) often live in a shared helper invoked from N pages/route handlers. When the helper's contract changes (e.g. `throw 401` → `redirect('/onboarding')`), the temptation is to update each consumer's error-handling block. Discipline: prefer one fix at the helper that mutates behavior at the source AND shrinks the surface (one branch instead of N), unless a per-consumer reason exists (different fallback UI, different telemetry tag). Why: per-consumer fixes drift over time as new consumers ship without remembering to update; helper-level fix is automatic for any future consumer. Distinct from "Sibling path symmetry" (line 132), which says fix-all-siblings-in-same-commit when the duplication can't be DRYed; this rule says DRY first, fix the surface, when the helper exists. Applies to any duplicated pattern across N callsites where a single helper would naturally own the contract. (source: superpowers-exec-tomi · project: kalori-mvp-stabilization)

- **Inverse-pill is the WCAG-safe CTA for dark-themed brands with a dominant non-passing brand color.** Dark bg (near-black) + dominant brand color (oxblood, crimson, dark-blue) often fails SC 1.4.3 (2.28:1 at `#8A2A1F` on `#0E0A08`). Flipping to brand-color-filled pill + light-text-on-dark-pill inverts the contrast (8.86:1 AAA) while preserving brand identity. Applies to primary + destructive CTAs. Document as WCAG tiebreaker over visual spec.
