## Failure diagnosis (Phase 7 protocol §c — Test infrastructure failure)

### Root cause

`.env.local` in the project root was generated by `vercel env pull` (modified `2026-05-12 01:45 +0700` — predates this batch, which started `2026-05-16T02:23Z`). Two issues compound:

1. **Wrong project:** `.env.local` carries **production** Supabase URL + keys (`dryysypycsexvlbabtwq`). The auth fixture's `resolveEnv()` falls back from `SUPABASE_TEST_*` to `NEXT_PUBLIC_*`, so without a separate `.env.test`, every spec runs against the project `.env.local` points at.
2. **Embedded `\r\n` in values:** `.env.local` values are quoted strings containing literal `\r\n` characters at the end (Windows artifact of `vercel env pull`). The Playwright `globalSetup` (`tests/e2e/fixtures/global-setup.ts`) strips outer quotes but does NOT strip the embedded `\r\n`. So the loaded `SUPABASE_SECRET_KEY` is something like `sb_secret_...\r\n` — Supabase's auth API rejects it with `Invalid API key`.

Both issues are independent of the library overhaul batch. Direct curl against the dev project's Supabase Auth API with the cleanly-formatted key from `Planning/devapikeys.txt` succeeds — verifying it's an env-loader artifact, not a Supabase-side regression.

### Workaround attempted

Wrote a temporary Node runner that loaded `Planning/devapikeys.txt` directly (clean, unquoted, no `\r\n`), mirrored values onto the `SUPABASE_TEST_*` aliases, and exec'd Playwright with that env. **Result:** auth fixture provisions the user successfully (admin.createUser returns 201), but the test still fails at the first `goto('/library')` — the request is redirected to `/login?redirect_to=%2Flibrary` because the **dev server (running with `.env.local` PROD creds) does not recognize the dev-project session cookie**.

Restarting the user's running dev server with dev creds was not done — it would interrupt the user's active session and is outside Phase 7's scope. The temporary runner was removed after the diagnosis.

### Recommended fix (out-of-band, NOT applied here)

Operator must restart the dev server (`pnpm dev`) with the DEV Supabase project env. Options, in increasing order of permanence:

1. **One-shot:** `cross-env NEXT_PUBLIC_SUPABASE_URL=... SUPABASE_SECRET_KEY=... pnpm dev` (using the clean values from `Planning/devapikeys.txt`).
2. **Per-test-run scaffold:** Create `.env.test.local` (Next.js merges this on `NODE_ENV=test`-ish runs) OR `.env.local.dev` and instruct the operator which one to symlink to `.env.local` for E2E runs.
3. **Permanent:** Fix `tests/e2e/fixtures/global-setup.ts` to (a) strip embedded `\r\n` from values, (b) prefer `.env.test` if present, (c) refuse to run if `NEXT_PUBLIC_SUPABASE_URL` points at the prod ref `dryysypycsexvlbabtwq`. Add a follow-up.

The dev-server restart is the only blocker; once it talks to the dev project, the auth fixture's session cookie will be recognized and the suite will execute.
