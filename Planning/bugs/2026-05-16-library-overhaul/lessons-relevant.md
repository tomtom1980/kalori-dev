- **[2026-05-08] `min-width: 0` cascade pattern is the canonical fix for "min-content overflows" in flex/grid containers; without it, a flex/grid child has implicit `min-width: auto` and refuses to shrink below its content's intrinsic width.** Required at EVERY level of the container chain — fixing only the top-level container leaves descendant overflows. Symptom set: at narrow viewports (mobile-375, tablet-768), `document.scrollWidth` exceeds `window.innerWidth` even though the outermost container has `max-width` clamps; child grids / flex rows that contain long text or wide content (charts, tables, code blocks) push the parent's intrinsic width past the clamp because their `min-width: auto` floors at content-intrinsic-width. Pattern: apply `min-width: 0` to (a) the flex/grid container itself, (b) every flex/grid descendant in the chain that holds shrinkable content, (c) any `display: grid` cell with a `1fr` track that holds text content. Pair with an objective overflow regression test — `expect(scrollWidth).toBeLessThanOrEqual(innerWidth)` at every breakpoint covered (mobile-375, mobile-414, tablet-768) — because pixel-diff visual baselines won't catch sub-viewport overflows that scroll horizontally but render visually-correct on capture. Bug #1 + Phase 7 regression of the mobile-ui-overhaul batch: REG-1 `/progress` mobile-375 (151px overflow), REG-2 `/dashboard` tablet-768 (124px), REG-3 `/progress` tablet-768 (30px) — all fixed by adding `min-width: 0` to descendants Bug #1's first pass missed. - **[2026-05-08] Framer Motion's `useReducedMotion` only reads the OS `prefers-reduced-motion` signal — apps with their own user-controlled reduce-motion toggles (localStorage flag / `data-attribute` on root / context provider) need a wrapper that ORs all signals; otherwise CSS `@keyframes` animations and Framer animations drift in their honoring of the toggle.** Symptom: user enables in-app "reduce motion" toggle → CSS-animated elements (gated on a `data-attr` matched by `[data-reduce-motion="true"] *` selector) freeze correctly; Framer-animated elements (gated on `useReducedMotion()` from `framer-motion`) keep playing because the framer hook only reads the OS media query. Pattern: wrap once in `lib/motion/defaults.ts` exporting an OR'd hook (e.g. `useReducedMotionApp()` returning `osPref || appPref`); subscribe via `MutationObserver` on `<html>`/`<body>` data-attribute + `StorageEvent` for cross-tab + same-tab `CustomEvent('reduce-motion-changed')` for reactivity within the same window. Consumers (LogFlowModal, WizardShell, MobileWheel*) flow through the wrapper; raw `useReducedMotion` is reserved for code paths that genuinely only care about OS preference (e.g. accessibility audit utilities). Two-baseline-test exception: helpers like `useReducedMotionVariants` invoked outside a component body can keep the raw hook and accept the OS-only limitation as a documented minor — see Phase 7's pending-minor-finding ledger. - **[2026-05-14] `prettier-plugin-tailwindcss` can silently strip a leading-space template-literal fragment, restoring a className-concatenation bug a fix had just resolved.** Symptom: writing `` className={`base-class${cond ? ' is-readonly' : ''}`} `` to gate a state modifier produces `base-classis-readonly` (one mashed token) after the pre-commit prettier hook runs — the leading whitespace inside the conditional fragment is collapsed, and CSS selectors `.base-class.is-readonly` never match. The bug-fix commit lands GREEN locally but the prettier hook silently re-introduces the original defect on commit. After every className-concat fix, run the formatter and verify the post-hook output preserves the fix — formatter hooks have opinions, and bug-fixes that depend on leading/trailing whitespace are exposure.
