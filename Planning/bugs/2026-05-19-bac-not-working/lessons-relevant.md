- **[2026-05-08] Codex R2 caught a fail-open daily-cap enforcement: `(computeDayTotalMl(...) ?? 0)` treats the helper's null-on-error return as "empty day," so any transient totals SELECT failure silently bypasses the entire 5000ml cap.** Generic principle: when a numeric reader returns `null` to mean "unknown" AND the consumer is a security/correctness gate (cap, rate limit, quota, eligibility check), `?? 0` (or any default-to-permissive) inverts the gate from fail-closed to fail-open during exactly the degraded-dependency conditions where you most need it to hold. Two compatible fixes: (a) distinguish error from null-data at the reader (`{value, ok}` shape) and have the consumer fail-closed (5xx) on `ok=false`; (b) move the read-and-act into a single atomic DB operation so there is no "unknown total" branch to default. The fail-safe-vs-fail-open principle (line 151) names the policy decision; this rule names the specific code-shape trap that violates it silently. Distinct from line 207 (catch-and-default with Sentry on read-side) — that's about visibility for graceful-degradation defaults; this is about correctness gates that must NEVER default-permissive. Applies to any "SELECT total/count/state, then enforce" pattern where the SELECT can fail. (source: bugfix-tomi · project: kalori)

- **[2026-05-09] Atomic per-user-per-day rate-limiting / cap enforcement pattern: `pg_advisory_xact_lock(hashtext(user_id::text || ':' || domain || ':' || date::text))` inside a `security invoker` Postgres function, paired with a custom SQLSTATE (e.g. `P0010` for `over_daily_limit`) and a numeric DETAIL field (e.g. `currentTotalMl::text`) the route handler parses back into the 409 body.** The advisory-lock key is per-(user, domain, date) so concurrent requests across different users / different domains don't serialize on each other; same-user same-day requests serialize through the lock until COMMIT/ROLLBACK releases it. `security invoker` keeps RLS active so the function can't be a privilege-escalation surface. The custom SQLSTATE lets the route handler distinguish cap-rejection from generic 5xx without string-matching the message. Mirrors the `library_merge_atomic` pattern in migrations 0008/0011 (Kalori convention). Implementation skeleton: `select pg_advisory_xact_lock(hashtext(...)); v_current := (select coalesce(sum(...),0) from <table> where user_id=v_user and date=p_date); if v_current + p_delta > p_cap then raise exception using errcode='P0010', detail=v_current::text, message='over_daily_limit'; end if; insert ...;` — all in one transaction. Pair with: `revoke all from public; grant execute to authenticated;` so anonymous callers cannot reach it. The route handler reads `rpcError.code === 'P0010'` and echoes `parseInt(rpcError.details, 10)` as `currentTotalMl` in the 409 body. Used in 0018_water_log_atomic_cap.sql for the daily-water-cap (5000ml). Generalizable to: per-user-per-day food-log calorie caps, per-user-per-hour AI request caps, per-user-per-day login-attempt caps, any "SUM-then-decide" enforcement where SUM-then-insert race is unacceptable. (source: bugfix-tomi · project: kalori)

- **[2026-05-09] Kalori canonical chip-state pattern (post 2026-05-09 water-FAB UX batch): mutation endpoints with chip/widget consumers return server-authoritative new state in the response body; clients commit that value directly rather than computing `current + delta` + invalidation guards.** `POST /api/water/log` returns `{ row, totalMl: number | null }` where `totalMl` is the SUM(ml) for the user-day post-write (computed via `mlFromWaterRow` over the row set indexed by `(user_id, date)`). The chip's success path commits `setCommittedConsumedMl(response.totalMl)` directly when `typeof totalMl === 'number'`, falling back to `current + ml` only when the server omits/nulls the field (rare aggregation read failure). Pattern eliminates the `resetKey` discriminator + `useLayoutEffect` ref-mirror complexity that ratcheted up across 3 Codex rounds before refactoring to server-authoritative; also collapses two symmetric failure modes (R1 double-count + R3 undercount) into one trivially-correct invariant. Optimistic-add reducer hardening (`if (delta.issuedResetKey !== state.resetKey) return state;`) is still needed for the OPTIMISTIC-PATH corner case where `useOptimistic` replays pending actions across a base-state change — the `issuedResetKey` discriminator continues to coordinate replay-vs-discard inside `useOptimistic`. When designing future Kalori mutation endpoints with chip/widget consumers (food-log entries, weight quick-add chip increments, micronutrient chips), include the new authoritative state field in the response shape from the start; server SUM-over-day-rows is bounded (≤ 12 rows per user-day for water, similar for food log) and the cost is a single indexed SELECT after the INSERT. (source: bugfix-tomi · project: kalori)

- **[2026-05-09] Server-side error fallbacks must fail loud — `try { ... } catch { return DEFAULT }` masks real defects; capture to Sentry (or equivalent) BEFORE the fallback returns or you'll never know the catch fired.** Codex R1 Critical C1 in the mobile-water-button batch caught a `(app)/layout.tsx` profile-timezone fetch where a column-name typo (`user_id` instead of `id`) made every Supabase query throw `PGRST116`-class errors, but the `catch` block silently returned `'UTC'` so non-UTC users had `loggedOn` derived from server-side UTC midnight — wrong-day writes in production with no signal. Fix added `Sentry.captureException(error, { tags: { component: 'layout', concern: 'timezone-derivation' } })` BEFORE the UTC fallback so the silent path becomes visibly noisy when it fires. Pattern: any catch block that returns a default value (timezone, locale, feature-flag default, "best guess" value) needs an explicit log/capture/metric at the catch site — even when the fallback itself is "graceful." Distinguish from line 197's `catch { swallow }` after a state commit (that's about state-machine consistency); this rule applies to read-side catch-and-default where silent failure produces wrong-but-rendered output. The fallback may still ship; it just must not be silent. Applies to RSC server fetches with default values, client-side `try { JSON.parse(...) } catch { return {} }`, any catch returning a "safe" default. (source: bugfix-tomi · project: kalori)

- **[2026-05-09] Kalori `profiles` table column for FK-mirroring `auth.users.id` is `id`, NOT `user_id` — copy from an existing call site rather than guessing.** Server-component layout fetches and any RSC-side `profiles` lookup must use `.eq('id', user.id)`; the column name `user_id` (which appears on other Kalori tables like `food_log_entries`, `weight_log`, `water_log`) does NOT exist on `profiles`. Codex R1 Critical C1 in the mobile-water-button batch caught a typo where `(app)/layout.tsx` used `eq('user_id', userId)` for a `profiles.timezone` SELECT — the query returned `PGRST204` (column does not exist) but a silent UTC catch hid the error from non-UTC users. Pattern: when adding a new `profiles`-table call site, grep an existing one (`app/(app)/dashboard/page.tsx` is a known-good example) and copy the column shape verbatim. Same pattern applies to other FK-mirroring tables (the Kalori convention is `id` when the row IS the user, `user_id` when the row REFERENCES the user). (source: bugfix-tomi · project: kalori)

- **[2026-05-16] Supabase / Postgres compare-and-set with `.lt(field, ceiling)` is NOT atomic — predicate must `.eq(field, preflight_value)` to pin the stale read so the loser's UPDATE affects 0 rows.** Under READ COMMITTED isolation, a `.lt('attempt_count', MAX)` predicate re-evaluates against post-first-update state, so a second concurrent UPDATE still matches the loose bound (`attempt_count=1 < 3`), still returns 1 row affected, and the caller wrongly believes it claimed the slot. The kalori sketch-pipeline batch's Round 1 "atomic UPDATE" fix was caught by Round 2 as still racing: both workers fired Gemini calls under realistic concurrent retries (double-click, two tabs, N requests within ms). Round 3 fix: `.eq('sketch_attempt_count', currentAttempts)` where `currentAttempts` is the value read in the same handler call — true compare-and-set, no advisory locks, no RPC, no extra columns. Pairs with the existing CAS/AtomicBoolean rule (line 158) but applies at the SQL predicate level rather than the in-process atomic primitive level. Applies to any read-check-write retry pattern in Supabase / Postgres: rate-limit counters, attempt-count slots, claim-state transitions, cap enforcement, optimistic-locking columns. Defense in depth: pair with an inline cap-check at the callsite, NOT just at the predicate, so the boundary holds even when the CAS shape changes. (source: bugfix-tomi · project: kalori)

- **[2026-05-16] [supabase] Incremental migration apply scripts MUST implement a 4-step idempotency strategy because `supabase_migrations.schema_migrations` is sparse / unreliable when prior cutovers used the Management API path (which doesn't update the tracker table): (1) probe the tracker for known-applied versions, (2) artifact-detection fallback — verify a specific higher migration's named artifact (table/column/index/function) exists via information_schema/pg_indexes SELECT, (3) explicit HALT with diagnostic output when tracker says "not applied" AND artifact-detection says "already there" (the two disagree → require user inspection, never silent-skip), (4) operator override flag (`--migrations 0018,0019,0020,0021` or `--only <slug>`) to bypass detection entirely when the operator already knows the truth.** Kalori E.1.2 incremental migration script: prior cutovers via Management API queries left the tracker empty, so a naive `SELECT version FROM supabase_migrations.schema_migrations` returned `[]` and the script wanted to apply every migration including the already-applied ones — silent re-apply of `0001..0017` would have failed loudly on conflicting CREATE TABLE statements but quietly succeeded on idempotent statements producing false-positive "successful" runs. Artifact-detection fallback (e.g. `does pg_indexes contain 'water_log_user_day_idx' yet?` to detect 0018 already applied) catches the tracker-emptiness case; explicit HALT on disagreement prevents the silent-skip-artifact-fallback hazard (artifact present + tracker absent could ALSO mean the operator manually dropped tracker rows to re-test, in which case re-apply IS desired). Pattern: every "apply pending migrations" script in any system with a non-trivial migration tracker (Supabase, Alembic, Flyway, Liquibase, Rails ActiveRecord schema_migrations) needs all 4 layers when the team has historically applied migrations through multiple paths. The operator-override flag is the escape hatch when detection produces a false signal. (source: superpowers-exec-tomi · project: kalori-mvp-stabilization)

- **[2026-05-01] Production deployment ≠ production migrations.** Vercel auto-deploys code from git on every push to main. Supabase migrations are NOT applied automatically. A "READY" Vercel deployment can run against a completely empty DB and silently 500 every authenticated route. Always verify DB schema state when claiming production-readiness, not just commit state — `select count(*) from migrations` or equivalent. Phase-close gates should explicitly include "production DB cutover applied" as a separate AC distinct from "production deployment READY." (source: superpowers-exec-tomi · project: kalori-mvp-stabilization)

- **[2026-05-14] When a new guard is added to a route handler with existing idempotency / replay / dedup logic, adversarial review (not TDD) is what catches the order-of-validation regression.** TDD covers the NEW contract; the EXISTING contract remains in code and stays GREEN against its own tests — but the new guard placed BEFORE the existing replay-lookup block silently rejects replays that should have succeeded. Symptom: TDD-RED-first for the new behavior all passes; the route's documented idempotency contract for `client_id` retry against an already-persisted row >30d ago now returns 400 instead of 200/replayed. Codex Round 2 caught this on Kalori C.5 only because the review framing was "verify the existing behavior didn't silently shift." Pattern: when adding ANY new rejection guard to a route handler with existing replay/idempotency/dedup logic, name the discipline explicitly in the Codex prompt — "verify each existing reject path AND each existing replay path still resolves correctly after the new guard lands." Pairs with line 38 (Fixing request SHAPE ≠ ERROR BEHAVIOR) — both rules name asymmetries where TDD's new-behavior focus and Codex's existing-behavior framing are non-substitutable. Reusable beyond idempotency: any time a guard is added to a multi-branch handler (auth gate, rate limit, quota, cap, eligibility check), classify EACH guard as "reject-all" vs "reject-new-only" and verify the placement against the existing branch graph. (source: superpowers-exec-tomi · project: kalori-mvp-stabilization)
