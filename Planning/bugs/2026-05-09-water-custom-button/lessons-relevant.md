# Relevant Lessons — Batch 2026-05-09-water-custom-button

## High-Signal for This Batch

- **[2026-05-09] Kalori canonical chip-state pattern (post 2026-05-09 water-FAB UX batch): mutation endpoints with chip/widget consumers return server-authoritative new state in the response body; clients commit that value directly rather than computing `current + delta` + invalidation guards.** `POST /api/water/log` returns `{ row, totalMl: number | null }` where `totalMl` is the SUM(ml) for the user-day post-write (computed via `mlFromWaterRow` over the row set indexed by `(user_id, date)`). The chip's success path commits `setCommittedConsumedMl(response.totalMl)` directly when `typeof totalMl === 'number'`, falling back to `current + ml` only when the server omits/nulls the field (rare aggregation read failure). Pattern eliminates the `resetKey` discriminator + `useLayoutEffect` ref-mirror complexity that ratcheted up across 3 Codex rounds before refactoring to server-authoritative; also collapses two symmetric failure modes (R1 double-count + R3 undercount) into one trivially-correct invariant. Optimistic-add reducer hardening (`if (delta.issuedResetKey !== state.resetKey) return state;`) is still needed for the OPTIMISTIC-PATH corner case where `useOptimistic` replays pending actions across a base-state change — the `issuedResetKey` discriminator continues to coordinate replay-vs-discard inside `useOptimistic`. When designing future Kalori mutation endpoints with chip/widget consumers (food-log entries, weight quick-add chip increments, micronutrient chips), include the new authoritative state field in the response shape from the start; server SUM-over-day-rows is bounded (≤ 12 rows per user-day for water, similar for food log) and the cost is a single indexed SELECT after the INSERT. (source: bugfix-tomi · project: kalori)
- **[2026-05-09] Defensive client-side guards have symmetric failure modes — a guard that prevents double-count via "skip on baseline shift" causes undercount when the shift is unrelated to the in-flight write; the canonical fix is a server-authoritative response, not better guards.** Round-1 Codex caught a chip-counter double-count race (in-flight POST + mid-flight `router.refresh()` baseline that already absorbed the same row → success-path commit added it again). R1's fix introduced a `resetKeyRef` discriminator that skipped the success-path commit when `resetKeyRef.current !== issuedResetKey`. Round-2 caught a microtask-vs-passive-effect timing hole in the ref mirror; R3 fix switched to `useLayoutEffect`. Round-3 then surfaced the SYMMETRIC failure: the discriminator drops successful writes whenever ANY baseline shift races mid-flight, including shifts UNRELATED to the in-flight write (e.g. another tab logs water, server-action triggers `router.refresh()` for a different reason). Both bugs share the same code surface — overcompensating with client-side guards is symmetric to the original bug. Canonical resolution: have the server return the new authoritative state (e.g. POST `/api/water/log` returns `{ row, totalMl }`) and have the client commit the response value directly; client-side prediction + invalidation guard is fundamentally fragile when the cache layer can shift orthogonally. Pattern: when the client is predicting state-after-mutation, prefer server-authoritative response shape over `predicted = current + delta` + reconciliation guards. Applies to any optimistic-update + invalidation flow (chip counters, list-pagination, dashboard widgets). Distinct from line 197's "silent data loss is always a bug" — this is about how to design the fix surface, not how to fail-loud. (source: bugfix-tomi · project: kalori)
- **[2026-05-09] Drill source values, not derived values — when a prop depends on "now" (today's date, current locale, current timezone), drill the INPUTS and recompute at use-time; precomputing in the parent and prop-passing captures the value at render time and goes stale across long-lived sessions.** Codex R2 Critical C2 in the mobile-water-button batch caught a `loggedOn` prop captured at server-render in `(app)/layout.tsx` and threaded through `<NavShell>` → after a session crossed local midnight, the FAB still POSTed yesterday's date. Fix drilled `timezone` (the input) instead of `loggedOn` (the derived value), and recomputed `userTzToday(timezone)` synchronously inside the click handler at tap time. Pattern: if `value = compute(input, now())`, drill `input` and call `compute(input, now())` at the moment of use; never drill `value`. Applies to any "today's date" / "current locale" / "active feature flag" / "current viewport class" prop on a long-lived component. The render-time-vs-use-time gap widens with session duration — short-session UIs may never observe it; long-lived dashboards / always-mounted nav chrome / PWAs in standalone mode reliably cross the gap. Pairs with line 264's `setTimeout`-after-await mountedRef pattern (also a render-time-vs-use-time gap) but applies to read-side derived values rather than write-side timer scheduling. (source: bugfix-tomi · project: kalori)
- **[2026-05-09] Kalori `profiles` table column for FK-mirroring `auth.users.id` is `id`, NOT `user_id` — copy from an existing call site rather than guessing.** Server-component layout fetches and any RSC-side `profiles` lookup must use `.eq('id', user.id)`; the column name `user_id` (which appears on other Kalori tables like `food_log_entries`, `weight_log`, `water_log`) does NOT exist on `profiles`. Codex R1 Critical C1 in the mobile-water-button batch caught a typo where `(app)/layout.tsx` used `eq('user_id', userId)` for a `profiles.timezone` SELECT — the query returned `PGRST204` (column does not exist) but a silent UTC catch hid the error from non-UTC users. Pattern: when adding a new `profiles`-table call site, grep an existing one (`app/(app)/dashboard/page.tsx` is a known-good example) and copy the column shape verbatim. Same pattern applies to other FK-mirroring tables (the Kalori convention is `id` when the row IS the user, `user_id` when the row REFERENCES the user). (source: bugfix-tomi · project: kalori)
- **[2026-05-09] Server-side error fallbacks must fail loud — `try { ... } catch { return DEFAULT }` masks real defects; capture to Sentry (or equivalent) BEFORE the fallback returns or you'll never know the catch fired.** Codex R1 Critical C1 in the mobile-water-button batch caught a `(app)/layout.tsx` profile-timezone fetch where a column-name typo (`user_id` instead of `id`) made every Supabase query throw `PGRST116`-class errors, but the `catch` block silently returned `'UTC'` so non-UTC users had `loggedOn` derived from server-side UTC midnight — wrong-day writes in production with no signal. Fix added `Sentry.captureException(error, { tags: { component: 'layout', concern: 'timezone-derivation' } })` BEFORE the UTC fallback so the silent path becomes visibly noisy when it fires. Pattern: any catch block that returns a default value (timezone, locale, feature-flag default, "best guess" value) needs an explicit log/capture/metric at the catch site — even when the fallback itself is "graceful." Distinguish from line 197's `catch { swallow }` after a state commit (that's about state-machine consistency); this rule applies to read-side catch-and-default where silent failure produces wrong-but-rendered output. The fallback may still ship; it just must not be silent. Applies to RSC server fetches with default values, client-side `try { JSON.parse(...) } catch { return {} }`, any catch returning a "safe" default. (source: bugfix-tomi · project: kalori)
- **[2026-05-08] When a bugfix-tomi batch ships design-change work alongside true bugs, the per-bug Codex review at Phase 4 must explicitly verify that any `Planning/ui-design.md` (or equivalent design-doc) edits land ATOMICALLY in the same commit as the implementation that ratifies them.** Bug #4 in the mobile-ui-overhaul batch self-added a tiebreaker (#23) to `Planning/ui-design.md` to justify expanding scope to LibraryTab's quantity-picker; the resume sub-agent verified the proposal had cited LibraryTab so the addition was warranted, but the pattern of self-justification-via-design-doc-edit is a known scope-creep risk — code that ships without the matching design-doc commit creates an undocumented pattern; design-doc that lands without the matching code creates an aspirational tiebreaker. Discipline: at Phase 4 Codex framing, name "design-doc edits MUST appear in the same commit-set as the implementation; flag any solitary `Planning/ui-design.md` change OR any new design pattern landing without a corresponding tiebreaker entry." This catches both directions: scope creep (tiebreaker self-added to ratify expansion the user didn't ask for) AND silent precedent (new design pattern shipped without paper trail). Pairs with the existing AC-text-vs-impl-reality lessons (lines 60, 164) — those are about ACs vs code; this is about design-doc vs code. (source: bugfix-tomi · project: kalori)

## TDD / Codex Discipline Reminders

- **[2026-05-08] False-green tests that assert PRESENCE (e.g., `getByTestId('trigger')` exists) without asserting BEHAVIOR (the trigger does its job) are technical debt with a green checkmark — Codex's adversarial framing "does this primitive serve its stated purpose?" catches them where TDD didn't.** TDD enforces "test must fail then pass," but the gap is in WHAT the test asserts: a test that mounts a component, asks for an element by testid, and verifies non-null can pass while the element does nothing user-observable when activated. C2 in the mobile-ui-overhaul batch shipped under TDD-green tests because the integration test only verified trigger render, not trigger behavior — Codex's R1 framing forced re-asking "what does this primitive PROMISE the user, and does the test exercise that promise?" Pattern at PR review: for every newly-added test that uses `getByTestId` / `getByRole` / `findBy*` without a follow-up `click()` + outcome assertion (state change, network call, focus shift, aria-live announcement), require either (a) the follow-up assertion or (b) a comment explaining why presence alone is sufficient (e.g. snapshot test, smoke test). Distinct from line 100's "live wire, dead consumer" pattern (boundary-write tests passing while consumer ignores the wire) — this is one layer earlier: the test never even probes the wire's behavior. (source: bugfix-tomi · project: kalori)
- **[2026-05-09] React `act()` flushes microtasks AND passive effects atomically — production microtask-vs-passive-effect race timing is invisible in any unit test under act(); pair source-level pin tests (asserting `useLayoutEffect` import / call usage) with E2E coverage that runs without act() to defend timing-sensitive code.** A `useEffect` mirroring a state value into a `useRef` does NOT close microtask-vs-passive-effect timing races: when an in-flight Promise's `.then()` continuation runs BEFORE React flushes passive effects, the ref reads stale. The water-FAB UX batch's R2 fix shifted to `useLayoutEffect` for synchronous timing — but the R3 regression test using `await waitFor(...)` MASKED the race because act() flushes microtasks AND passive effects in the same atomic step, so any test asserting "ref is current at success-callback time" passes whether the impl uses `useEffect` (passive) or `useLayoutEffect` (sync). Pattern: defense-in-depth for ANY timing-sensitive ref/effect requires (a) a source-pin test that grep-asserts `useLayoutEffect` is imported AND called against the specific state-key, (b) E2E coverage at the production composition that doesn't run inside act(), so the real microtask-vs-passive-effect ordering is exercised. Applies to any "live-read state inside an async closure that outlives the render" pattern. Distinct from line 87 ("Snapshot fields BEFORE `advanceUntilIdle`") and line 100 ("live wire, dead consumer") — this is one layer further: act() makes all effect-timing variants behaviourally indistinguishable in unit tests. (source: bugfix-tomi · project: kalori)

## Mobile / Animation Library Discipline

- **[2026-05-08] Framer Motion's `useReducedMotion` only reads the OS `prefers-reduced-motion` signal — apps with their own user-controlled reduce-motion toggles (localStorage flag / `data-attribute` on root / context provider) need a wrapper that ORs all signals; otherwise CSS `@keyframes` animations and Framer animations drift in their honoring of the toggle.** Symptom: user enables in-app "reduce motion" toggle → CSS-animated elements (gated on a `data-attr` matched by `[data-reduce-motion="true"] *` selector) freeze correctly; Framer-animated elements (gated on `useReducedMotion()` from `framer-motion`) keep playing because the framer hook only reads the OS media query. Pattern: wrap once in `lib/motion/defaults.ts` exporting an OR'd hook (e.g. `useReducedMotionApp()` returning `osPref || appPref`); subscribe via `MutationObserver` on `<html>`/`<body>` data-attribute + `StorageEvent` for cross-tab + same-tab `CustomEvent('reduce-motion-changed')` for reactivity within the same window. Consumers (LogFlowModal, WizardShell, MobileWheel*) flow through the wrapper; raw `useReducedMotion` is reserved for code paths that genuinely only care about OS preference (e.g. accessibility audit utilities). Two-baseline-test exception: helpers like `useReducedMotionVariants` invoked outside a component body can keep the raw hook and accept the OS-only limitation as a documented minor — see Phase 7's pending-minor-finding ledger. (source: bugfix-tomi · project: kalori)
- **[2026-05-08] Framer Motion's `animate={{ y: 0 }}` writes `transform: translateY(...)` directly, OVERWRITING any CSS-baked `transform: translate(-50%, -50%)` centering on the same element.** Symptom: a centered modal / overlay element styled with CSS `position: fixed; top: 50%; left: 50%; transform: translate(-50%, -50%)` jumps to the top-left corner the instant Framer's `animate` prop applies because the inline style takes precedence. Pattern fix: SEPARATE the centering element (CSS owner) from the animator element (Framer owner) — wrap `<m.div animate={...}>` in a static centering parent that owns the CSS `transform: translate(-50%, -50%)`, OR animate via a CSS variable that composes with translate (e.g. `--y` CSS var → `transform: translate(-50%, calc(-50% + var(--y)))` — Framer animates the var, CSS retains the centering). The wrapper-parent option is simpler; the CSS-var option keeps the animator on the centered element if the structure must stay flat. Discovery cost: silent on hover/idle, surfaces only when the animation runs; visual baseline tests catch it only if the captured frame is mid-animation. (source: bugfix-tomi · project: kalori)
