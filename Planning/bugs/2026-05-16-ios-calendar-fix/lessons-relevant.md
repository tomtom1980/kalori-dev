- **[2026-05-14] `<input type="datetime-local">` minute truncation creates a client/server boundary asymmetry: any minute-precision UI lower-bound + millisecond-precision server upper-bound (or vice-versa) needs a grace buffer OR a re-derive-on-submit pattern.** Pattern that bites: client formats a frozen `nowAtMount - 30d` clamp to `datetime-local` (minute precision — seconds dropped via floor); server-side bound recomputed with fresh `Date.now() - 30d` at request time. Network latency + minute-slice + modal-open drift means the displayed `min` is up to ~150s STALER than the server's bound; user clicks the displayed minimum, posts, server rejects for boundary reasons. Symptom appears only under realistic latency / mount-to-submit delay, not in zero-latency unit tests. Two compatible fixes: (a) server-side grace buffer (`BACKFILL_GRACE_MS`) subtracted from the server bound — narrow enough to not silently extend the documented window by hours, wide enough to cover all realistic drift sources (minute-trunc ~59s + modal-open ~90s + network ~few hundred ms → ~4 minutes typical worst case); (b) re-derive the lower-bound on the client at submit-time and post the value alongside the timestamp so the server validates against the client-derived bound. Tightening the client display to "honest" minutes is wrong — it lies to users about the available window. Applies to ANY `datetime-local` lower-bound + ms-precision server check, AND symmetrically to upper-bound clamps (future-time guards). Add a defense-in-depth integration test asserting "30d + 5min still rejected" to prove the grace is narrow. (source: superpowers-exec-tomi · project: kalori-mvp-stabilization)

- **[2026-05-08] Fixture extensions that paper over a server contract violation MASK a production bug; the real-POST E2E that exposed the bug is the right surface to keep — log a Critical followup, don't silently absorb.** When extending an E2E auth fixture to make a real-POST AC pass (e.g. flipping `timezone` from Asia/Ho_Chi_Minh to UTC to dodge a `400 date_in_future` server error caused by UTC-midnight comparison ignoring the user's offset), the test workaround is a productivity fix BUT the underlying server bug exists for production users in the offending timezone band. Discipline: (a) the fixture extension ships in the same commit as the failing AC, (b) a Critical followup is filed naming the production-impact (e.g. `F-B4-DATE-CONTRACT-TZ-AWARE` — east-of-UTC users saving local 00:00–06:00 get rejected), (c) once the server contract is fixed, the fixture flip reverts and the original-tz E2E case is restored as a regression. Mocked unit tests would have HIDDEN this entirely; the real-POST E2E EXPOSED it — keep the real-POST shape, never collapse to mock-and-forget. (source: superpowers-exec-tomi · project: kalori-mvp-stabilization)

- **[2026-05-08] False-green tests that assert PRESENCE (e.g., `getByTestId('trigger')` exists) without asserting BEHAVIOR (the trigger does its job) are technical debt with a green checkmark — Codex's adversarial framing "does this primitive serve its stated purpose?" catches them where TDD didn't.** TDD enforces "test must fail then pass," but the gap is in WHAT the test asserts: a test that mounts a component, asks for an element by testid, and verifies non-null can pass while the element does nothing user-observable when activated. C2 in the mobile-ui-overhaul batch shipped under TDD-green tests because the integration test only verified trigger render, not trigger behavior — Codex's R1 framing forced re-asking "what does this primitive PROMISE the user, and does the test exercise that promise?" Pattern at PR review: for every newly-added test that uses `getByTestId` / `getByRole` / `findBy*` without a follow-up `click()` + outcome assertion (state change, network call, focus shift, aria-live announcement), require either (a) the follow-up assertion or (b) a comment explaining why presence alone is sufficient (e.g. snapshot test, smoke test). Distinct from line 100's "live wire, dead consumer" pattern (boundary-write tests passing while consumer ignores the wire) — this is one layer earlier: the test never even probes the wire's behavior. (source: bugfix-tomi · project: kalori)

- **[2026-05-08] E2E `waitForTimeout(N)` after a fire-and-forget POST is always wrong — replace with `waitForResponse(predicate)` registered BEFORE the action.** Fixed-duration sleeps after a POST that round-trips through a cross-region pair (e.g. Vercel `iad1` ↔ Supabase `ap-southeast-1` ≈ 150–200ms one-way RTT, plus server processing) flake under contention even when the value (e.g. 500ms) is "comfortably" larger than median latency. The deterministic pattern: `const undoResponse = page.waitForResponse(r => r.url().includes('/api/...') && r.request().method() === 'POST' && r.status() === 200); await page.click(...); await undoResponse;` — registers the listener BEFORE the click so the response can't fire before `waitForResponse` is attached, and asserts the actual contract (URL match + method + status) rather than a wall-clock proxy. Same pattern beats `waitForTimeout` in any scenario where the test's logical "next step" is "wait for this server side-effect to land" — undo POSTs, optimistic-update reconciliations, mutation-then-RSC-revalidation flows. The cost is one extra line; the savings are zero flakes under cross-region latency, parallel-worker contention, or CI cold-start. Bug 2 + Bug 3 of the e2e-regressions batch were both this exact failure mode — bulk-delete-undo and single-delete-undo flaked on `waitForTimeout(500)` and went 100% green on `waitForResponse(...)` after 30 reps. (source: bugfix-tomi · project: kalori)

- **[2026-05-08] `setTimeout` scheduled in an async success branch fires on a destroyed component unless guarded at BOTH schedule site AND callback.** Unmount cleanup that clears the *currently-pending* timer ID does NOT prevent a NEW timer being scheduled later when `await fetch()` resolves AFTER the user navigates away. The defer pattern (`onSuccess → setTimeout(() => router.refresh(), 200)`) lands `router.refresh()` on whatever route the user is on now. Pattern: `mountedRef` checked before scheduling the timer AND inside the timer callback; `useEffect` cleanup sets `mountedRef.current = false`. Same trap with `requestAnimationFrame`, `queueMicrotask`, and any post-await side effect. (source: superpowers-exec-tomi · project: kalori-mvp-stabilization)

- **[2026-05-08] `app/(app)/loading.tsx` (Next 16 RSC nav skeleton) resolves Playwright's `page.goto()` BEFORE a server-side `redirect()` commits — synchronous `expect(page.url()).toContain('/login')` flakes; switch to `await page.waitForURL(/<pattern>/)`.** A protected route with both a server `redirect()` (e.g. `redirect('/login?reason=...')` from an auth fence) AND a sibling `loading.tsx` boundary paints the skeleton during the redirect commit window; Playwright sees the navigation as "complete" against the pre-redirect URL while the client-side router is still committing the new URL. Probe-spec evidence: post-fix dashboard `goto` resolves immediately at `/dashboard` (skeleton painted), then within ~ms the URL flips to `/login?reason=session_expired&redirect_to=%2Fdashboard`. Synchronous `page.url()` reads the pre-flip URL and the test fails despite the production redirect contract being correct. Pattern: replace `expect(page.url()).toContain('/login')` with `await page.waitForURL(/\/login\?.*redirect_to=/, { timeout: 5000 })` (matches the actual asserted contract: redirect happened with redirect_to query param). Same trap on any RSC redirect path under a `loading.tsx` boundary — auth fences, role-gated pages, feature-flag bounces. Distinct from `waitForResponse` for fire-and-forget POSTs: this is for navigation/redirect commit timing, not server round-trip timing. (source: bugfix-tomi · project: kalori)

- **[2026-05-15] Bounding-box y-coordinate comparison is the layout-relative assertion that survives semantically-opaque framework wrappers — when a component sits inside `<FadeUpCard>`, `motion.div`, Suspense boundaries, or other transparent containers, DOM source-order + selector-chain assertions are brittle.** Pattern: `const [a, b] = await page.evaluate(() => [document.querySelector(selA).getBoundingClientRect().y, document.querySelector(selB).getBoundingClientRect().y]); expect(b).toBeGreaterThan(a);` (or use Playwright's `locator.boundingBox()` per element + assert `box.y` ordering). Defensive `expect(box).not.toBeNull()` first so a paint regression surfaces with a readable assertion error rather than a TypeScript null-deref. Used on Kalori C.E2E.1 AC3 ("Micros panel renders below Macros panel") — neither panel has a top-level testid; bounding-box ordering held the assertion stable through `FadeUpCard` motion wrapping that would have broken any source-order or descendant-selector probe. Applies to any "is element B visually below element A" / "is panel C to the right of panel D" assertion where intermediate framework chrome obscures DOM topology. Distinct from line 124 (computed-style sibling-equality for tokens) — that's about resolved values matching a sibling; this is about LAYOUT POSITION ordering across opaque containers. (source: superpowers-exec-tomi · project: kalori-mvp-stabilization)
