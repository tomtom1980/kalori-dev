What this app is

Kalori is a private AI-assisted calorie and nutrition tracker built as a Next.js app. The package config describes it as an “AI-first calorie/nutrition tracker,” using Next.js, React, TypeScript, Tailwind, Supabase, Gemini, Vercel, and Sentry . The root layout loads three UI fonts — Newsreader, Inter, and JetBrains Mono — and sets the app metadata to “Kalori” / “AI-first calorie & nutrition tracker” .

At a high level, the app lets a user:

Sign in with magic link or Google.
Complete onboarding to calculate nutrition targets.
Log meals by typing, taking/uploading a photo, choosing from their personal library, or entering manually.
Review/edit AI-parsed nutrition before saving.
Track calories, macros, micronutrients, water, weight, and progress.
Build a personal food library automatically.
Get weekly AI review text when enough data exists.
Authentication and routing

The app uses Supabase Auth. Middleware protects private routes and redirects unauthenticated users to /login, while public routes are allowed through. Signed-in users visiting /login get redirected to /dashboard, unless the session has expired .

The login page supports email magic link and Google OAuth. The server page resolves the origin and passes it to the client login form . The client form validates email input, calls supabase.auth.signInWithOtp, and also supports signInWithOAuth({ provider: 'google' }) .

The /auth/callback handler exchanges the Supabase code for a session, then checks profiles.onboarding_completed_at. New/incomplete users go to /onboarding; completed users go to /dashboard or a safe same-origin redirect_to target .

App shell and navigation

All authenticated pages are wrapped by app/(app)/layout.tsx, which loads the current user ID and renders NavShell .

NavShell provides the main app chrome: sidebar, top bar, mobile bottom tab bar, mobile log FAB, keyboard shortcut handling, a lazy-mounted log-flow modal, undo toast mount, screen-reader live regions, and a skip-to-main link .

So the app is not just a set of pages; it has a persistent authenticated shell with a global “log food” modal available from multiple entry points.

AI features

There are three main Gemini-backed AI routes.

Text meal parsing

POST /api/ai/text-parse accepts user text, validates it, authenticates, sanitizes it, computes a per-user cache key, checks cache, calls Gemini if needed, validates the AI response with Zod, writes cache/cost logs, and returns parsed nutrition. On AI failure, it returns a fallback payload so manual logging can continue .

Vision/photo meal parsing

POST /api/ai/vision mirrors the text route but accepts base64 image data, enforces a 500 KB decoded-size limit, supports image MIME types, hashes image content for cache, calls Gemini Vision, validates output, logs cost, and falls back safely on failure .

Weekly review

POST /api/ai/weekly-review generates or retrieves a weekly review. It requires a Monday week_start_on, rejects future weeks, aggregates food entries for the week, skips Gemini if there are fewer than three logged days, persists the review into weekly_reviews, and logs the AI call exactly once per logical request .

Food entry saving and undo

POST /api/entries/save is the core meal-save endpoint. It validates food items, meal category, source, optional library item, and AI reasoning. It enforces auth, prevents future timestamps, resolves the user’s timezone, checks ownership of referenced library items, implements idempotency via client_id, inserts the food entry, optionally creates a library item, and invalidates dashboard/progress cache tags .

DELETE /api/entries/[id] deletes a user-owned entry, resolves the original user-timezone day for cache invalidation, and is used by the undo flow after a saved entry .

Personal food library behavior

The library has exact-name dedup preflight via POST /api/library/dedup-check, which checks the authenticated user’s own food_library_items for a matching normalized_name .

Bulk delete is implemented as a tombstone/undo flow. POST /api/library/bulk-delete soft-deletes up to 100 items by setting deleted_at, then invalidates the library cache . POST /api/library/bulk-delete/undo clears deleted_at for matching original row client_ids if still within the undo window .

Merging duplicates is implemented through POST /api/library/merge. It validates winner/loser IDs, prevents self-merge, calls a Postgres RPC library_merge_atomic, repoints entries from loser to winner, updates the winner, deletes the loser, and invalidates affected library/entry/progress caches .

Water and weight tracking

Water quick-add posts to POST /api/water/log, which validates glass, bottle, or ml, stores the row in water_log, uses client_id for idempotency, and invalidates the user’s day entry cache .

Weight quick-add posts to POST /api/weight/log. It validates date/weight/note, rejects future or older-than-30-day dates, inserts into weight_log, and may update the profile’s BMR/TDEE/calorie target when auto recalculation is needed .

Reviewer B — Database/API contract
Critical issue B1 — Manual entries likely fail at DB level

The confirmation flow can produce source: "manual" when manual fallback is submitted. LogFlowTabs converts manual fallback into a confirmation payload with source: 'manual' .

The save API also accepts source: z.enum(['text', 'photo', 'library', 'manual']) in app/api/entries/save/route.ts .

But the food_entries table migration shown defines the source check as only ('text','photo','library'), excluding manual .

Expected failure: Manual fallback save reaches /api/entries/save, passes Zod, then DB insert fails with a check-constraint error and the route returns db_error.

Fix options:

Preferred:

alter table public.food_entries
drop constraint if exists food_entries_source_check;

alter table public.food_entries
add constraint food_entries_source_check
check (source in ('text','photo','library','manual'));

Alternative: do not use source: "manual"; map manual fallback to source: "text" or another existing DB value. I would not recommend that because you lose useful source information.

Critical issue B2 — “Save to library” probably never creates a library item

In ConfirmationScreen, the save body sets save_to_library = true when the toggle is enabled, but I do not see it sending normalized_name .

The API only creates a food_library_items row if all of these are true:

body.save_to_library
body.source !== 'library'
body.normalized_name

That condition is in /api/entries/save .

So the UI can show “save to library,” but the API condition is not satisfied because normalized_name is missing.

Impact: New AI-parsed foods are saved as food entries, but they do not get added to the personal food library.

Best fix: Compute normalized name on the server instead of trusting the client.

In /api/entries/save, when save_to_library is true:

const normalizedName =
  body.normalized_name ?? normalizeName(body.items[0]?.name ?? '');

Then insert if normalizedName is non-empty.

Also include more complete nutrition when saving library items. Current insertion only stores { kcal }, which makes future library re-log/detail quality weaker .

High issue B3 — Tombstone/delete code relies on deleted_at, but baseline schema lacks it

Library fetch performs a lazy tombstone sweep using deleted_at, then selects only rows where deleted_at IS NULL .

Single-item delete sets deleted_at . Bulk delete also sets deleted_at . Library update filters .is('deleted_at', null) .

But the inspected food_library_items migration does not create deleted_at .

Impact if no later migration exists/applied: Library page, update, delete, undo, and lazy sweep can fail with “column deleted_at does not exist.”

Fix: Add migration:

alter table public.food_library_items
add column if not exists deleted_at timestamptz;

create index if not exists food_library_items_user_deleted_idx
on public.food_library_items (user_id, deleted_at);
High issue B4 — AI idempotency expects ai_call_log.client_id, but baseline schema lacks it

lib/ai/cost-log.ts writes client_id into ai_call_log when provided and uses findPriorCall() to query .eq('client_id', input.clientId) .

The migration shown for ai_call_log contains id, user_id, call_type, input_hash, tokens, cost_estimate, latency_ms, cached_flag, and created_at, but no client_id .

The code comments reference a later migration, but I could not confirm it from the inspected files.

Impact if missing in the real DB:

Replay detection fails.
AI route retries can call Gemini again.
logAICall() insert with client_id can error, but it swallows most insert errors after Sentry capture, so the user may not see this directly.

Fix migration:

alter table public.ai_call_log
add column if not exists client_id uuid;

create unique index if not exists ai_call_log_user_client_unique_idx
on public.ai_call_log (user_id, client_id)
where client_id is not null;
Medium issue B5 — Dedup check does not ignore tombstoned rows

/api/library/dedup-check selects matching food_library_items by user_id and normalized_name, but does not filter deleted_at IS NULL .

Impact: If a user deletes a library item and then logs the same food before the lazy sweep hard-deletes it, the dedup prompt can suggest reusing a deleted item.

Fix:

.is('deleted_at', null)

Add that to the dedup query.

Reviewer D — AI/photo/storage
Medium issue D1 — Thumbnail upload likely fails for many photo logs

SnapTab compresses the selected image once and checks that the compressed image is under 500 KB before calling the vision endpoint . The compression helper default is also 500 KB / 1600 px .

Then SnapTab sends that same base64 to /api/storage/thumbnail .

But /api/storage/thumbnail rejects anything over 50 KB .

The upload failure is swallowed in SnapTab, so meal parsing still works, but thumbnails often will not be saved.

Fix: Generate two outputs:

Vision image: ≤500 KB, ≤1600 px
Thumbnail image: ≤50 KB, ≤320 px, ideally WebP/JPEG

Example direction:

const visionImage = await compressImage(file, {
  maxSizeBytes: 500 * 1024,
  maxWidthOrHeight: 1600,
});

const thumbnailImage = await compressImage(file, {
  maxSizeBytes: 50 * 1024,
  maxWidthOrHeight: 320,
  mimeType: 'image/webp',
});

Then send visionImage to Gemini and thumbnailImage to storage.

Fix: Remove the inline outline: 'none', or add an explicit class with a :focus-visible style. Best:

// remove outline: 'none'

Then let the global :focus-visible rule apply.
